Technical specification for Codex: repurpose the existing sticky-notes site into a Nintendo Switch skill-building games board

Project intent

Take the existing static sticky-notes website that currently presents Seattle reference content and repurpose it into a new content site about Nintendo Switch games that can support useful real-life skills, habits, or forms of practice. This is not a redesign from scratch. It is a content and structure conversion built on top of the existing implementation, interaction model, and visual language.

The current site already has the right foundation: a tactile sticky-notes board, semantic category sections, searchable notes, filtering, bookmarking, expandable details, and a build-free HTML/CSS/JS architecture. The task is to preserve what is strong about that foundation, remove the Seattle-specific content and assumptions, and rebuild the site around a much richer editorial model for games.

The new site should not feel like an online store, a gaming blog, or a generic review directory. It should feel like a curated reference board for discovering games that are interesting because they may support practice, learning, creativity, coordination, planning, reflection, or recovery. The emphasis is on useful mechanics and credible framing, not hype, fandom, or popularity.

Core product goal

The final result should be a static website that lets a user browse, search, and filter a large collection of Nintendo Switch games through the lens of practical value. Each game should be presented as a note on the board. The user should be able to quickly understand what kind of activity the game involves, what kind of real-life skill or useful habit it may support, how confident the site is in that connection, and where to go for the official product information.

This site should help people discover overlooked or unusually valuable titles, not just obvious bestsellers. Some of the strongest candidates may be niche tools, rhythm games, coding-like games, builder games, movement games, or titles that structure practice in a way that ordinary entertainment games do not. The site should make those distinctions visible and useful.

At the same time, the site must remain honest. It should not imply that a game automatically improves a person's life. It should describe concrete activities clearly, suggest plausible usefulness where justified, and signal when the evidence is stronger or weaker. The site should feel edited by a careful human, not generated by a storefront scraper.

What Codex is starting from

The working directory already contains a complete Seattle-themed version of the site. That version includes the HTML shell, CSS, JavaScript, content source files, icons, and a deep research report in both Markdown and PDF. Codex should treat the current Seattle site as the implementation base and the deep research report as the main editorial source for the new game-focused content.

The previous Seattle content should no longer appear in the live site when this task is complete. That includes the page title, hero copy, categories, links, and note bodies. The implementation patterns may remain, but the content identity must change completely.

The final result must still be build-free. Do not introduce a framework, package manager, generator, or server dependency. The site should remain easy to open locally and easy to maintain by editing source files directly.

Overall editorial direction

The most important editorial shift is this: the site is not about "good games" in general. It is about games that have some concrete connection to useful practice, structured thinking, creativity, coordination, physical activity, calm routines, or similar real-world value.

That means Codex must not fill the board with generic action games and then attach vague claims such as "improves strategy" or "improves reflexes." That kind of loose wording is exactly what should be avoided. The board should favor titles where the relationship between game activity and useful practice is visible, specific, and explainable.

A strong entry is one where the player is clearly doing something that resembles a meaningful task or drill. Examples include building systems, solving logic constraints, composing music, learning programming ideas, drawing, maintaining movement routines, following rhythm precisely, coordinating body motion, or practicing focused repetition. A weak entry is one where the usefulness claim is mostly wishful language pasted onto a normal game loop.

Codex should therefore think like an editor, not just a collector. The point is not to maximize the number of titles regardless of quality. The point is to build a large but credible collection, where each note earns its place.

Required content transformation

The Seattle notes must be replaced with game notes. Each note should represent one game. Each board section should represent one useful skill area or practice area rather than a city-service category.

The board should no longer be organized around geography or city services. It should instead be organized around meaningful patterns of activity. Good categories are categories that help a user reason about why a game belongs there. A category should not simply mirror traditional game genre labels such as action, RPG, or platformer unless that genre label also reflects a useful practice mode, which in most cases it does not.

A strong category system for the first version would include creative and maker-oriented games, logic and systems-thinking games, movement and fitness games, rhythm and timing games, calm and low-stress routine games, and more cautious brain-training or drill-style games. Codex may adjust the exact category names if better wording emerges during implementation, but the structure must stay skill-oriented rather than genre-oriented.

The category design should make the board easy to scan. A user should be able to think, "show me games that help with structured thinking," or "show me games that are useful for movement and routine," and then immediately find relevant notes.

What a game note should communicate

Every note should do a lot of work quickly. At a glance, the user should be able to see the game title, an image, a short summary, the main useful skill or practice angle, and a visible evidence or confidence marker. The note should not be overloaded, but it should not be vague either.

The front of the note should contain only the most important information. It should answer these questions quickly, without literally printing the questions themselves.

First, what kind of activity does the game actually involve? The summary should describe the mechanic in plain terms, not in marketing language.

Second, why might that activity be useful or interesting outside the game? This should be phrased conservatively and specifically. Good wording sounds like "supports repeated practice in timing and coordination" or "provides hands-on exposure to logic and rule-building." Bad wording sounds like "will make you smarter" or "boosts life skills."

Third, how strong is the confidence behind the note? The site needs a visible evidence marker so the user can separate stronger entries from more speculative ones.

The details area of the note should go deeper, but it should still read as natural prose. Do not turn the expanded content into a stack of stiff subheadings like "What you do," "Why it matters," "Evidence note," and so on. That style feels artificial and clumsy. Instead, write two or three short, well-ordered paragraphs. The first paragraph should explain the actual play pattern. The second should explain the plausible practical value or limit of that pattern. A third short paragraph may add release context, caveats, or evidence framing where helpful. Then present the links in a small utility block at the end.

The note should read like a compact editorial annotation, not like a template dump.

Evidence model

The site should make it easy to distinguish between stronger and weaker claims without making the interface academic or heavy. A visible evidence badge is the right approach, but the badge must be explained somewhere on the page in plain language.

A simple three-level model is appropriate. The strongest level should mean that the game's useful value is supported by a clear mechanic and reasonably strong external grounding or obvious transfer. The middle level should mean the connection is plausible and meaningful, but more interpretive. The weakest level should mean the game is still interesting for this board, but the real-world value claim is more tentative, situational, or disputed.

The badge should never be used to imply scientific certainty where none exists. It is an editorial confidence signal, not a scientific rating system. The user should understand that some games are on the board because they are genuinely useful, while others are there because they create an interesting kind of practice, but with more uncertainty about broader transfer.

This evidence model should also be filterable. A user should be able to narrow the board to only the stronger entries if desired.

Category system and board structure

The old Seattle categories must be fully removed from the live board. The new board should use categories that reflect useful activity types.

A strong first-pass structure would begin with creative and maker games. This is likely to be one of the richest sections because it can include music tools, drawing tools, game-building tools, design-oriented games, and flexible sandbox construction titles.

A second major section should focus on programming, logic, automation, and systems thinking. This is where games that involve formal reasoning, automation chains, logic constraints, or computational problem-solving should live.

A third section should cover movement, exercise, and body routine games. These are especially important because they offer a concrete real-world practice connection and can be described clearly without overclaiming.

A fourth section should cover rhythm, precision, and coordination. Some of these games may overlap with the creative section, but the board should still allow them to be found through the coordination lens.

A fifth section can cover calmer or low-stress games that may help with routine, recovery, or decompression, but this section must be written carefully and should avoid medicalized or universal claims.

A sixth section can cover explicit brain-training or drill-oriented titles, but only if Codex treats that space skeptically and clearly indicates that far-transfer claims are often mixed or limited.

If a seventh category emerges naturally during implementation, such as planning and management or language and communication, Codex may add it, but only if there is enough strong material to justify a separate section. Empty or filler categories should not exist.

Hero section rewrite

The hero area should be completely rewritten for the games project. Remove all Seattle-specific copy. The new hero should introduce the site as a curated Nintendo Switch board for games with practical, creative, cognitive, physical, or routine-building value.

The hero should also help orient the user to the editorial point of view. It should explain, briefly and cleanly, that the board prioritizes useful mechanics, honest evidence framing, and discovery of overlooked titles. It should not talk about implementation details, source formats, or the fact that the page is hand-authored.

A small summary of the board's scope is appropriate in the hero. For example, it may show how many categories are present, how many games are included, and how many entries have stronger evidence support. Those metrics should be relevant to the board rather than decorative.

The tone should be practical and confident. It should sound like a serious, well-edited reference page, not like a game magazine intro and not like a software readme.

Control bar changes

The current site already has a useful control bar with search, category filtering, bookmarks, reset, and expand/collapse behavior. That should remain, but it needs to be extended and tuned for the new editorial model.

Category filtering must remain. In addition, Codex should add evidence filtering so that users can quickly narrow the board to stronger or more tentative entries. If space and complexity allow, Codex may also add a skill-type or tag-based filter, but evidence is the more important second filter.

Search must index more than game titles. It should include note summaries, tag values, evidence descriptions, and the deeper prose in the expanded details. A search for terms like "coding," "memory," "music," "fitness," or "logic" should produce meaningful results without requiring exact title matches.

Bookmarks should remain, because this kind of discovery board benefits from personal curation. Expand and collapse actions should also remain because the new notes will likely contain richer details than the Seattle sample.

The control area should stay compact and disciplined. Do not let it grow into a cluttered dashboard.

Images and media handling

The Seattle board was largely text-based. The game board needs representative imagery, but imagery should support the notes rather than dominate them.

Each game note should include one representative image. Prefer official cover art, product key art, or a strong official screenshot. The image must be correct for the exact game entry and should be verified carefully. It must not be a fan image, unrelated sequel art, incorrect platform art when a platform-specific version matters, or an image pulled blindly from a noisy search result.

Codex should prefer official Nintendo store art where possible, then official publisher or developer assets, then other reliable sources only if necessary.

Images must be downloaded locally and normalized. Do not leave the final site dependent on hotlinked external images. Do not keep arbitrary source dimensions. Create one consistent asset strategy and apply it to all game images. One consistent aspect ratio, one predictable display pattern, and modest file sizes are more important than preserving every original crop.

The images should live in a dedicated local folder. CSS can scale them for display, but the underlying assets should already be reasonably optimized. The board needs to stay performant even with a substantial number of notes.

Attribution must be present for images, but it must remain quiet. A small attribution line or compact source link is enough. It should be visible, accurate, and unobtrusive. The user asked specifically for attribution that exists without becoming distracting. That means it should probably live in the details area or in a very small image footer rather than in the main headline region of the note.

Links and outbound destinations

Every game note should include a canonical official destination. The best default is the official Nintendo store listing if it exists for the target platform and region being used as the reference. This gives the user a stable product page and provides consistent release context.

Where an official game site exists and is useful, include it as a secondary link. If the developer or publisher has a particularly useful product page, that can also be included. Avoid low-value link clutter. The links area should feel curated.

If a game has an especially relevant evidence source, review source, or research-adjacent source that helps justify the note's usefulness claim, that may be included as a secondary or tertiary reference, but only when it adds real value. The note should not become a mini bibliography.

Data collection process Codex should follow

Codex should not try to scrape the entire idea from the open web in one blind sweep. The better approach is staged.

First, use the deep research report as the initial seed set. It already contains strong anchor titles, category suggestions, and a useful evidence framing model. That report should determine the early structure of the board and the first wave of included games.

Second, validate those games with current official sources. Confirm title spelling, release context, official store pages, official sites, and usable image sources. This stage is about cleaning and grounding the seed set.

Third, expand selectively. Use the seed games and their tags as leads for discovering adjacent titles, but be strict. A game should only be added when there is a clear reason it belongs on the board. The site should not become a giant pile of barely relevant entries just because related titles exist.

Throughout this process, Codex should keep a clear internal distinction between three things: the actual mechanic of the game, the plausible usefulness of that mechanic, and the strength of support behind that usefulness claim. These are not the same thing, and collapsing them would degrade the quality of the board.

Writing style for note content

The prose needs to feel natural, compact, and informative. Avoid AI-sounding scaffolding, repetitive structure markers, and stiff labels. Expanded note content should read like concise editorial writing.

A good note says what the player actually does, why that pattern may be useful, and how confident the site is about that usefulness, all without visibly forcing those thoughts into named subfields.

The style should stay skeptical of marketing language. Words like "immersive," "innovative," and "engaging" are usually useless here unless they refer to a concrete design property. Focus on what the player practices, repeats, or learns.

Also avoid the opposite problem: do not become dry to the point of being unreadable. The board should still feel inviting and curious. It should reward browsing.

Required code and file changes

The HTML must be reworked so that the page title, hero copy, control copy, category sections, and note content all reflect the Nintendo Switch skill-board concept rather than Seattle. The existing semantic structure is worth preserving where it still fits, because it already supports a good board layout and the current interaction model.

The CSS must be adapted to support image-bearing notes, evidence badges, denser metadata, and a board that can hold many more notes without feeling visually chaotic. The sticky-note identity should remain, but the new board likely needs tighter note structure, more disciplined image slots, and slightly more consistent spacing.

The JavaScript must be updated to support the new filtering model, especially evidence filtering. Existing features such as bookmarks, search, and expanded details should remain and should work with the new content model. Search must index the new prose and metadata, not just visible note titles.

The content source strategy should also improve. Even if the final site is still plain HTML, Codex should maintain a cleaner source file for the games content so future updates are easier. The old Seattle content file should no longer be treated as the live editorial source for the board. Codex may choose a Markdown or JSON-based content source if that helps maintenance, but the finished website must remain a simple static site.

Visual direction for the repurposed board

The site should still feel like a board of physical notes, not like a digital storefront. Keep the paper-like surfaces, layered composition, and tactile note metaphor. But this repurposed version should be more structured and more disciplined than the Seattle sample, because the information density is higher and the notes now include images.

The board should become slightly more systematic in how notes are sized and arranged. There can still be variety, but the variety should not make it difficult to compare notes or skim a section. Images, metadata, and badges should line up in a way that feels intentional.

Color can help category scanning, but the palette should remain restrained. The site should feel curated and mature, not toy-like.

The image should act as a fast anchor. The title, summary, and usefulness angle should still do most of the work. This is not a cover-art gallery.

Performance matters. If the number of notes grows substantially, the design should stay smooth and readable. Keep image sizes under control and do not add gratuitous visual cost.

What good output looks like

When the task is done, the Seattle site should be gone from the live experience. In its place there should be a Nintendo Switch discovery board that feels like it belongs to the same family of interfaces, but is clearly built for a richer and more editorially demanding subject.

A user landing on the page should immediately understand that this is a curated collection of Switch games with practical or skill-related value. They should be able to filter the collection by category and by evidence level, search within it, bookmark games, and open note details for deeper context.

The notes should look good, feel readable, and include images without losing the sticky-note identity. The page should feel coherent and handcrafted.

The content should feel selective and trustworthy. Even when a game is included with weaker evidence, the note should explain that carefully rather than pretending certainty. That editorial honesty is one of the core values of the project.

Implementation expectations

Codex should not stop at a planning document. It should actually perform the repurposing work. That includes editing the site files, replacing the Seattle content, collecting game data, gathering and normalizing images, wiring the new filters, and delivering a working local result.

The first complete pass does not need to exhaust the universe of all possible games, but it should produce a substantial and convincing catalog, not a placeholder shell. The structure should also be clearly designed for future expansion.

Codex should leave behind one short implementation notes file that explains what content structure it chose, how image attribution is handled, how evidence filtering works, and what remains open for future growth.
