---

A00 Specification Summary

---

This project creates a browser bookmarklet that transforms the main readable content of the current web page into an AI-chat-style streaming presentation. The bookmarklet does not generate new text, call an AI service, summarize, rewrite, or change the article's meaning. It extracts the existing main article content from the page, temporarily clears or replaces that main content region, then re-renders the same content back into the page with a staged typing effect and polished visual treatment inspired by modern AI assistant interfaces.

The implementation will be authored in modern JavaScript and bundled with Bun.js into a single JavaScript file. Bun is assumed to be installed on the system. The final bookmarklet payload should be readable, not mangled, and not minified. The bookmarklet URL itself will start with `javascript:`, but the bundled source file does not need to include that prefix. Instead, the installation page will dynamically construct the full bookmarklet URL.

The bundle will expose a named top-level function called `BookmarkletMain`. The installation page, `index.html`, will load the generated bundle as a normal script. Once loaded, the page will call `BookmarkletMain.toString()` and dynamically construct a bookmarklet URL in the form:

```js
javascript:(<BookmarkletMain source>)()
```

The installation page will then render a draggable bookmarklet link. The user can drag this link into the browser bookmarks bar. When clicked on an article page, the bookmarklet runs entirely in the current page context.

The bookmarklet may use the provided readability-related libraries, or it may copy only the required extraction logic into the project. The intended extraction goal is to identify the primary article or readable content on arbitrary websites. Libraries such as Mozilla Readability are suitable because they are designed to isolate the main content from surrounding navigation, ads, sidebars, comments, and other page chrome. The final behavior should prioritize preserving readable article content over perfect visual reconstruction of the original site.

The experience should feel like the page is being "answered back" by an AI chat interface. The original article text appears to vanish from the main reading area, then returns progressively, paragraph by paragraph, sentence by sentence, and token by token. The animation should suggest streaming output: subtle cursor movement, soft fades, minor line settling, gentle blur recovery, controlled pauses, and block-level rhythm changes. It should look deliberate, smooth, and technically serious, not like a novelty typing demo.

The user scenario is recreational but the implementation quality should be production-minded. A user is reading an article, activates the bookmarklet for amusement, and watches the article reappear as if an AI assistant is composing it live. The interaction should not destroy page state permanently, should not lose content, should not make text unreadable, and should include a clear way to stop or restore the original content.

---

B00 Product Definition

---

The product is a single-browser bookmarklet called, for example, "AI Stream Article". Its purpose is to apply a temporary AI-chat-style streaming visual effect to the main readable content of the current page.

The bookmarklet is not a reader mode replacement, although it borrows extraction behavior from reader-mode libraries. It is not an AI client, not a scraper, not a summarizer, and not a content editor. Its value is experiential: it takes existing page content and presents it through the visual language of modern AI response interfaces.

The main user action is simple. The user visits a web page with an article or long-form readable text, clicks the bookmarklet in the bookmarks bar, and the article's main text region is transformed into a streaming response surface. The same text is replayed with AI-like pacing and visual effects.

The visual result should feel like a blend of ChatGPT, Claude, Copilot, and Gemini-style response rendering without copying any vendor's exact branding, logos, names, protected visual identity, or claims of actual AI generation. The bookmarklet may use phrases such as "Streaming article" or "Replaying page content", but it should not claim the content was generated by ChatGPT or any other service.

---

C00 Goals

---

The primary goal is to create a reliable bookmarklet that can run on many websites and identify the main readable content with high practical accuracy.

The second goal is to preserve content integrity. The bookmarklet must not lose paragraphs, headings, inline links, emphasized text, code blocks, blockquotes, or list content when those elements are part of the main article. The rendered replay should remain readable at all times.

The third goal is to create a convincing AI-streaming visual experience. The effect should be detailed enough to feel intentional: text appears progressively, blocks settle into place, a cursor or active token indicator moves naturally, and the pacing varies based on content structure.

The fourth goal is to produce a clean developer workflow. Bun.js will bundle the source into one JavaScript file. The bundle should remain readable. It should avoid minification, avoid mangling, and be easy to inspect.

The fifth goal is to provide a polished installation page. The page should explain what the bookmarklet does, show concise usage instructions, and render the final draggable bookmarklet link dynamically from the loaded bundle.

---

D00 Non-Goals

---

The bookmarklet will not call remote AI APIs.

The bookmarklet will not send page content to a server.

The bookmarklet will not summarize, rewrite, translate, censor, rank, or interpret the article.

The bookmarklet will not permanently alter the page.

The bookmarklet will not attempt to bypass paywalls, login walls, anti-scraping protections, DRM systems, or site access controls.

The bookmarklet will not exactly imitate or reproduce the proprietary UI of any named AI product. It will create a generic AI-chat-style streaming experience.

The bookmarklet will not guarantee perfect extraction on every website. Sites with unusual DOM structures, heavy shadow DOM, canvas-based text, or aggressive content security policies may require fallback behavior.

E00 Build And Packaging Requirements

---

The project will use Bun.js as the build tool. The source code should live in a small, understandable project structure.

A practical structure is:

```txt
project-root/
  package.json
  bun.lockb
  src/
    bookmarklet.js
    article-extractor.js
    renderer.js
    effects.js
    styles.js
    controls.js
  public/
    index.html
  dist/
    bookmarklet.bundle.js
```

The Bun build command should bundle `src/bookmarklet.js` into `dist/bookmarklet.bundle.js`.

The build should preserve readability. The generated JavaScript should not be minified and should not mangle function names.

The bundle must expose `BookmarkletMain` in a way that works when included by `index.html`. Because the installation page will call `BookmarkletMain.toString()`, the function must be available on the global `window` object after the bundle script loads.

A suitable output pattern is:

```js
function BookmarkletMain() {
  // Full bookmarklet implementation or embedded bootstrap.
}

window.BookmarkletMain = BookmarkletMain;
```

The final bookmarklet URL should be constructed by the installation page, not hardcoded in the bundle.

The bookmarklet code should be self-contained when converted to a string. This has an important consequence: if `BookmarkletMain.toString()` is used directly, every helper function needed at runtime must either be nested inside `BookmarkletMain`, serialized into the generated string, or otherwise included in the constructed bookmarklet expression. The specification recommends one of two approaches.

The first approach is to author normal modular source files, then have Bun bundle them into a single file that assigns one complete function to `window.BookmarkletMain`. This function should contain all runtime helpers inside its own function body.

The second approach is to let the install page construct the bookmarklet URL from a named bootstrap function plus serialized helper code. This is more complex and less desirable for this project.

The recommended approach is the first one: `BookmarkletMain` should be the self-contained runtime function.

F00 Installation Page Requirements

---

The installation page will be a static `index.html` file. It will include the generated bundle using a script tag:

```html
<script src="../dist/bookmarklet.bundle.js"></script>
```

After the script loads, the page will dynamically create the bookmarklet URL from `BookmarkletMain.toString()`.

The page should render a clear title, a short description, installation instructions, compatibility notes, and a prominent draggable bookmarklet button.

The dynamic rendering logic should create an anchor element whose `href` is the bookmarklet URL. The visible label can be "AI Stream Article" or another concise name.

The installation page should not store or transmit any page content. It only prepares the bookmarklet link.

The page should show manual installation instructions for browsers that prevent drag-and-drop bookmarklet installation. The manual flow is: create a new bookmark, set its name, copy the generated bookmarklet URL, and paste it into the bookmark's URL field.

The page should include a copy button for the generated bookmarklet URL. Since bookmarklet URLs can be long, the UI should display a shortened visual preview while copying the full value to the clipboard.

The installation page should explain that the bookmarklet works best on article pages, documentation pages, essays, blog posts, and long-form pages with a clear main content area.

The installation page should explain that the bookmarklet runs locally in the current tab and replays existing text; it does not use AI and does not send article content anywhere.

G00 Bookmarklet Runtime Flow

---

When the user clicks the bookmarklet, `BookmarkletMain` runs immediately in the current page.

The first runtime step is to check whether the bookmarklet is already active. If an active instance exists, the second click should toggle the experience. Depending on current state, it should either stop the animation, restore the original content, or show a compact control panel with options.

The second step is to locate the main readable content. The bookmarklet should attempt extraction using a hierarchy of methods. The strongest method is a Readability-style parse against a cloned document. The next fallback is a DOM scoring algorithm that identifies likely article containers using tag names, text density, heading density, link density, paragraph count, and semantic elements such as `article`, `main`, and `[role="main"]`. The final fallback is to use the largest text-dense visible container on the page.

The third step is to preserve the original content. Before changing anything, the bookmarklet should store the target element's original HTML, inline style attributes that it changes, scroll position, and any metadata needed to restore the page.

The fourth step is to transform the target element into a controlled replay surface. This means the target element is cleared or visually replaced, then filled with a generated container that will receive streamed content.

The fifth step is to normalize extracted content into a sequence of renderable blocks. Blocks may include headings, paragraphs, lists, blockquotes, code blocks, images with captions, tables, and horizontal separators.

The sixth step is to animate those blocks into the page. The renderer should stream textual content progressively while preserving structure.

The seventh step is to mark completion. When all content has replayed, the active cursor should disappear or settle, the final content should remain readable, and a small unobtrusive control should allow restore, replay, pause, resume, and speed adjustment.

H00 Article Extraction Requirements

---

The extraction system should favor the actual article body over navigation, comments, sidebars, cookie banners, subscription prompts, related links, and footer content.

The preferred extraction path is to use a Readability-style parser. The parser should operate on a cloned document so the original page remains untouched during extraction. The parsed result usually includes title, byline, excerpt, content HTML, text content, site name, and length.

If a readability library is bundled, the project should evaluate bundle size and bookmarklet URL length. Bookmarklets can become difficult to install when URLs are extremely long. If the library makes the bookmarklet too large, the project may copy or implement only the extraction portions needed for this use case.

If the parser returns usable content, the bookmarklet should map the parsed result back to the original page when possible. Replacing the original article container is preferable to injecting a detached overlay because it preserves scroll context and makes the experience feel like the article itself is transforming.

If mapping back to the original element is unreliable, the bookmarklet may render an overlay reader surface centered over the page. This overlay should still feel connected to the page, with a subtle translucent backdrop and a restore button.

The extraction logic should consider these candidates first: `article`, `main`, `[role="main"]`, `.article`, `.post`, `.entry-content`, `.content`, `.story`, `.markdown-body`, `.prose`, and text-dense containers with multiple paragraphs.

The extractor should reject candidates with high link density, very short text, mostly hidden content, excessive button/input elements, or text dominated by navigation labels.

The extractor should preserve the article title. If the title is present in the extracted content, it should stream as the first heading. If it is not present, the bookmarklet may inject the parsed title above the content.

The extractor should preserve meaningful inline links. Links should appear in the streamed content as normal anchors, but their text should stream progressively like other text. The final link should remain clickable.

The extractor should preserve code blocks and preformatted text. Code should not be tokenized as natural language if that harms readability. Code blocks may stream line by line or appear with a faster "terminal fill" effect.

The extractor should preserve blockquotes. They should receive a distinct streaming treatment: a soft vertical rule, slight inset, and slower reveal to suggest quoted emphasis.

Tables should be handled cautiously. A table should usually fade in row by row rather than token by token because token-level table streaming can break alignment and readability.

Images that are part of the article should not be removed. They may fade into place at the moment their surrounding block appears. Captions may stream as text below them.

---

I00 Content Integrity Requirements

---

The transformed article must contain the same user-visible article text as the extracted source.

Text must not be reordered.

Paragraph boundaries must be preserved.

Headings must remain headings.

Lists must remain lists.

Code must remain code.

Blockquotes must remain blockquotes.

Links must remain links after streaming completes.

The bookmarklet may add visual wrappers, status labels, controls, cursors, and progress indicators, but these additions must be clearly separate from the article content.

The bookmarklet must not permanently modify the original DOM. It should either store and restore the original HTML or render into a reversible wrapper.

The restore action must be available during and after the animation.

If the extraction fails, the bookmarklet should not destroy content. It should show a small non-invasive message: "Could not identify a main article on this page."

---

J00 Visual Experience Principles

---

The visual experience should communicate three things at once: the page is being transformed, the content is being streamed, and the article remains readable.

The first moment after clicking the bookmarklet is important. The article should not abruptly disappear into a blank area. Instead, the current content should soften, dim slightly, blur by a few pixels, and collapse into a calm transition state. This transition should be brief, approximately 250 to 450 milliseconds. It should feel like the page is handing control to a response renderer.

The replay surface should look like a focused answer area. It should not look like a modal advertisement, not like a popup, and not like a developer console. The ideal look is a clean reading container with subtle rounded corners, soft background separation, a faint border, and a barely visible glow near the active line.

The active text should stream with controlled irregularity. Real AI streaming does not appear as a perfect mechanical typewriter. It arrives in small bursts, pauses slightly after punctuation, moves faster through common phrases, slows at headings, and sometimes completes a sentence with a quick final burst.

The active block should have a transient visual state. Newly appearing text may start at very slight opacity reduction, a 1 to 2 pixel downward offset, or a tiny blur. It should settle immediately into crisp text. The effect should be subtle enough that the user never struggles to read.

The visual language should avoid excessive animation. No shaking, flashing, large particle effects, rainbow gradients, or aggressive cursor blinking. The desired tone is polished, modern, and restrained.

---

K00 Detailed Streaming Animation Model

---

The renderer should convert content into a sequence of blocks. Each block has a type, a source node, text fragments, child inline elements, and an animation profile.

Paragraphs should stream token by token. A token can be a word, punctuation mark, or whitespace group. Words appear in small bursts. Punctuation can trigger a pause. Paragraphs should start with a gentle fade-in container, then fill with text.

Headings should appear more ceremonially. The heading container fades in first, then the heading text appears in larger chunks. A level-one heading may reveal in 2 to 5 phrase segments rather than word by word. This avoids making titles feel tedious.

Short paragraphs should stream slower than their length might imply because they often carry emphasis. Long paragraphs should stream at a practical pace. The renderer should not make the user wait several minutes for a long article unless the user chooses a slow speed.

Lists should reveal item by item. Within each item, text may stream token by token. The bullet or number marker should appear before the item text begins.

Blockquotes should reveal with a slow left-border glow, then text should stream at a reflective pace. The quote should feel intentionally set apart from normal paragraphs.

Code blocks should use a different rhythm. They can reveal line by line, with a faint monospace cursor effect. Very long code blocks should accelerate after the first few lines.

Tables should render structure first, then fade rows into visibility. The header row appears first. Body rows appear in sequence. Cell text should not stream token by token unless the table is very small.

Images should fade in with a very slight scale-up from 0.985 to 1. Captions should stream after the image is visible.

The cursor should be a small inline mark at the current generation point. It can be a vertical bar, a rounded pill, or a soft glowing dot. It should follow the active token and disappear after completion.

The animation should support pause and resume. Pausing should freeze the current block without losing progress. Resuming should continue from the same token.

The animation should support skip to end. This action immediately renders all remaining content in its final state.

---

L00 Timing And Pacing Requirements

---

The default speed should be fast enough for amusement without becoming frustrating. The target default should make a medium article feel animated and alive, while completing in a reasonable time.

A practical default is 35 to 70 tokens per second, adjusted by content type. This is faster than normal human typing, closer to AI streaming perception. The animation should use burst groups, not one timer per character.

Punctuation should affect pacing. A comma may add a short pause. A period may add a longer pause. A paragraph break may add a noticeable pause. A heading may add a small dramatic delay before the next block.

The renderer should support at least three speed modes: relaxed, standard, and fast. Relaxed emphasizes the effect. Standard is the default. Fast preserves the streaming style but minimizes waiting.

The renderer should dynamically accelerate for long articles. If the article exceeds a configured threshold, such as 5,000 words, the bookmarklet should increase burst size and reduce pauses after the first few paragraphs. The experience should remain recognizable without forcing the user through a long replay.

The renderer should avoid layout thrashing. It should batch DOM updates using `requestAnimationFrame` and append document fragments where possible.

The animation loop should stop cleanly if the user restores the page, navigates away, or clicks the bookmarklet again.

---

M00 Visual Effects Specification

---

The bookmarklet should use a layered set of subtle effects.

The first effect is the content handoff. The original article fades to approximately 20 to 35 percent opacity, blurs slightly, and then clears. The blur should not exceed 4 pixels. The transition should be short.

The second effect is the response surface. The article area becomes a clean container with a background that adapts to the page. On light pages, it can use a near-white translucent surface. On dark pages, it can use a dark translucent surface. The surface should have a soft border and faint shadow.

The third effect is active-line emphasis. The current block being streamed can receive a faint background highlight or edge glow. This highlight should fade away once the block is complete.

The fourth effect is token reveal. New text appears with a very short opacity ramp and no distracting motion. For example, each burst can be inserted at full layout position with opacity transitioning from 0.2 to 1 over 80 to 140 milliseconds.

The fifth effect is cursor presence. The cursor should be visible only while streaming. It should pulse gently but not blink aggressively.

The sixth effect is block settling. When a block completes, its active glow fades out and its text becomes visually indistinguishable from normal article text.

The seventh effect is ambient progress. A very thin top progress line or small status pill can show that replay is underway. It should be optional and visually quiet.

The eighth effect is completion. At the end, the cursor fades, the progress line completes, and the control pill changes from "Streaming" to "Complete". No celebration animation is required.

The ninth effect is restore transition. If the user restores the original content, the streamed surface fades out and the original content fades back in. The user should not feel that the page has been damaged.

The tenth effect is reduced-motion support. If the user has `prefers-reduced-motion: reduce`, the bookmarklet should minimize transitions and stream content in larger chunks with fewer animated effects.

---

N00 Layout And Styling Requirements

---

The bookmarklet should inject its own CSS using a style tag with a unique ID, such as `ai-stream-bookmarklet-styles`.

All class names should be namespaced, for example `asb-root`, `asb-surface`, `asb-block`, `asb-cursor`, and `asb-controls`. This reduces the chance of conflicting with page CSS.

The replay surface should inherit the article's approximate width where possible. It should not force the article into an excessively narrow chat bubble unless the original layout is too wide for comfortable reading.

The content should be readable on both light and dark pages. The bookmarklet should detect computed background and text color, then choose a compatible surface theme. If detection fails, it should use neutral colors and inherit the page font.

The bookmarklet should preserve the original article typography where practical. It can improve spacing, but it should not make the article look unrelated to the page.

The renderer should avoid using fixed global fonts unless necessary. For body text, inherited page font is acceptable. For controls and status labels, a system UI font is acceptable.

The control panel should be small and attached near the top-right of the replay surface or fixed in the lower-right corner if the article container position is unreliable.

The controls should include Restore, Pause or Resume, Replay, Speed, and Skip. These controls should not obscure article text.

The bookmarklet should use a high z-index only for controls and optional overlay mode. It should not blanket the page unnecessarily.

---

O00 Accessibility Requirements

---

The final article text must be selectable.

Links must be keyboard-accessible after rendering.

The control buttons must be keyboard-accessible.

The control panel should have clear text labels, not icon-only controls.

The animation must respect `prefers-reduced-motion`.

The active streaming region may use `aria-live="polite"` only if it does not create an overwhelming screen reader experience. Because streaming every token to a screen reader would be disruptive, the recommended behavior is not to announce every token. Instead, announce status changes such as "Article replay started", "Article replay paused", and "Article replay complete".

Text contrast must remain adequate. The bookmarklet should not apply low-contrast gradients or heavy transparency that makes the article hard to read.

The cursor should not rely on color alone. Its shape and position should communicate the active point.

---

P00 State Management Requirements

---

The bookmarklet should maintain state on `window`, using a unique key such as `window.__AI_STREAM_BOOKMARKLET_STATE__`.

The state should include whether the bookmarklet is active, the selected target element, the original HTML, the original inline styles changed by the bookmarklet, the current animation controller, the extracted content model, current speed mode, pause state, and cleanup functions.

Running the bookmarklet while active should not create a second renderer. It should open or focus the existing controls, or it should toggle restore depending on the chosen behavior.

The restore function should remove injected bookmarklet DOM, remove injected styles if no longer needed, restore the original target content, cancel timers, cancel animation frames, and clear global state.

The animation controller should use an abort signal or equivalent cancellation flag to stop all asynchronous loops safely.

---

Q00 Error Handling Requirements

---

If no main content is found, the bookmarklet should show a small message without modifying the page.

If extraction succeeds but rendering fails, the bookmarklet should restore the original content and show a concise error message.

If the current page blocks inline bookmarklet execution due to browser or site restrictions, the bookmarklet may fail silently at the browser level. The installation page should mention that some sites may prevent bookmarklets from running.

If the bundle is too large to work comfortably as a bookmarklet in some browsers, the project should document the limitation and consider trimming extraction code.

If the page has a complex shadow DOM, the bookmarklet should attempt normal document extraction first and then fall back to visible text density. It does not need to deeply traverse every shadow root in the first version.

If the article is extremely long, the bookmarklet should still render it, but it may switch to faster block-based streaming.

---

R00 Security And Privacy Requirements

---

The bookmarklet must run locally in the current tab.

It must not make network requests.

It must not inject remote scripts.

It must not send article content anywhere.

It must not store article content in local storage, session storage, cookies, IndexedDB, or remote storage.

It may keep temporary content in memory only while the bookmarklet is active.

The installation page should not require analytics, external fonts, remote CSS, or third-party scripts.

The generated bookmarklet URL should be inspectable. Because the bundle is not minified, the user should be able to review the implementation.

S00 Detailed User Experience Flow

---

The user opens an article page and begins reading normally. The page looks untouched.

The user clicks the bookmarklet from the bookmarks bar.

The article area responds within a fraction of a second. The surrounding page remains in place, but the main content softens as if the reading surface is being prepared.

A compact status pill appears near the article: "Preparing article".

The original main content fades and clears. The replacement surface appears in the same region.

The title appears first. It does not type letter by letter. It resolves in a few deliberate phrase chunks.

The first paragraph begins streaming. Words arrive in short bursts. The active cursor rests at the end of the latest visible token.

The user sees subtle motion but can read continuously. Nothing jumps, flashes, or becomes illegible.

As paragraphs complete, they settle into normal article text. The active glow moves to the next block.

If the article contains a quote, the quote appears inset. Its left rule brightens slightly, then the quote streams more slowly than regular paragraphs.

If the article contains code, the code block appears as a contained monospace region. Lines reveal quickly, almost like a terminal filling in.

If the user clicks Pause, the cursor freezes and the status changes to "Paused".

If the user clicks Resume, streaming continues from the same place.

If the user clicks Skip, the rest of the article appears immediately in final readable form.

If the user clicks Restore, the original article DOM returns.

When the replay completes naturally, the status changes to "Complete", the cursor fades out, and the controls remain available.

T00 Usage Scenario 1: The Curious Reader

---

The user is halfway through a long essay. The page is quiet, mostly white, with a dense column of text. They click the bookmarklet not because they need help, but because they want to see the page behave differently.

The article pauses visually. The words they were reading soften, as if the page has taken a breath. The text does not explode or vanish abruptly. It recedes for a moment, then the same reading area becomes a calm response surface.

The title returns first. It arrives in confident fragments, not as individual letters. The user recognizes the article immediately. Then the first paragraph begins to form. The cursor sits at the edge of each new burst, and the text settles into crisp focus almost as soon as it appears.

The user feels that the article has become active. It is no longer a static page. It feels like the page is speaking its own contents back in a controlled, AI-like cadence.

---

U00 Usage Scenario 2: The Late-Night Blog Post

---

The user is reading a personal blog at night in dark mode. The page already has a quiet mood. They press the bookmarklet.

The bookmarklet detects the dark background and chooses a darker replay surface. The surface does not glare. It adds a soft border and a subdued shadow, enough to separate the article from the rest of the page without turning it into a modal.

The first paragraph streams with a warm rhythm. Short sentences arrive quickly. Longer sentences slow slightly near commas and periods. The cursor glow is visible but restrained.

The user watches the article rebuild itself. The effect feels like an assistant composing a thoughtful response, but the content is still exactly the blog post. The mood stays intact. The bookmarklet enhances the reading moment without overpowering it.

---

V00 Usage Scenario 3: The Technical Documentation Page

---

The user is reading technical documentation with headings, paragraphs, inline code, and code blocks. They click the bookmarklet.

The extractor identifies the main documentation content, not the sidebar navigation. The left navigation remains visible or visually secondary, while the main documentation column transforms.

The heading appears first. Then explanatory text streams in a normal paragraph rhythm. Inline code appears as part of the sentence and remains visually distinct.

When a code block appears, the behavior changes. The block frame fades in, then lines populate quickly from top to bottom. The effect resembles a terminal receiving output, but it remains clean and readable. The bookmarklet does not type every character of a long code sample slowly because that would frustrate the user.

The user sees the documentation as a generated answer while still retaining the structure that makes documentation usable.

---

W00 Usage Scenario 4: The News Article

---

The user opens a news article with a headline, byline, image, caption, and several paragraphs. They click the bookmarklet.

The bookmarklet isolates the story body and avoids the header menu, newsletter signup, related story widgets, and comment prompts. The headline returns in a composed reveal. The byline may appear as a small metadata line if it was part of the extracted readable article.

The lead image fades back into place before the caption streams underneath it. The first paragraph then begins. It arrives in bursts that feel fast but not rushed.

The user perceives the article as if it were being delivered through a modern AI response pane. The surrounding site still exists, but the main article has become a performance.

---

X00 Usage Scenario 5: The Essay With Quotations

---

The user is reading an essay with several quoted passages. They trigger the bookmarklet.

Normal paragraphs stream at standard speed. When the first quotation appears, the renderer slows slightly. A vertical rule fades in beside the quote. The quoted text appears with a calmer cadence, as though the interface is giving the passage extra weight.

After the quote completes, the active glow leaves the quote and returns to the next normal paragraph. The user can visually distinguish the author's voice from quoted material.

The effect makes the structure of the essay feel more theatrical without changing the text.

---

Y00 Usage Scenario 6: The Very Long Article

---

The user activates the bookmarklet on a very long feature article. The first moments feel detailed: title, opening paragraph, and first section stream with full AI-like pacing.

After enough content has appeared, the bookmarklet quietly accelerates. Burst sizes grow. Pauses after punctuation shorten. Whole sentences sometimes arrive together. The active cursor remains present, so the experience still feels like streaming, but the user is not forced to wait too long.

The user can continue watching, pause, skip to the end, or restore. The system respects the user's time while preserving the visual concept.

---

Z00 Usage Scenario 7: The Failed Extraction

---

The user clicks the bookmarklet on a page that is mostly an app dashboard, not an article.

The bookmarklet scans the page but does not find a reliable main reading container. It does not clear the page. It does not guess recklessly. Instead, a small message appears near the corner: "Could not identify a main article on this page."

The user closes the message and continues using the page normally. The bookmarklet has failed safely.

---

AA00 Usage Scenario 8: The Reader Who Changes Their Mind

---

The user starts the replay and enjoys it for a few seconds, then decides they want the original article back.

They click Restore. The active cursor stops. The replay surface fades down. The original article content returns in place. The scroll position remains close to where the user was.

The user does not feel trapped inside the effect. The bookmarklet behaves like a reversible layer, not a destructive page modification.

---

AB00 Usage Scenario 9: The Reduced-Motion User

---

The user has reduced motion enabled at the operating system level. They trigger the bookmarklet.

The bookmarklet detects the preference and uses larger, calmer content reveals. Instead of many small token animations, paragraphs appear in larger chunks with minimal transition. The cursor is less animated. Blur and glow effects are reduced.

The user still gets the idea of streamed content, but the page avoids unnecessary motion.

AC00 Usage Scenario 10: The Installation Experience

---

The user opens the static installation page. The page explains that the bookmarklet replays the main article text with an AI-style streaming effect.

A button appears with the bookmarklet name. The user drags it to the bookmarks bar. The page also provides a copy option for manual bookmark creation.

The installation page generated the bookmarklet dynamically from the bundled `BookmarkletMain` function. The user does not need a build server, browser extension, account, API key, or remote service.

The bookmarklet is now installed as a normal browser bookmark.

AF00 Implementation Detail: Extraction Fallback Scoring

---

If a readability parser is unavailable or fails, the bookmarklet should score candidate elements.

The score should increase with paragraph count, visible text length, average sentence length, semantic tag value, heading proximity, and ratio of text to markup.

The score should decrease with high link density, many navigation-like links, many buttons, many form controls, hidden or offscreen content, repeated short labels, and class names suggesting sidebar, nav, menu, footer, comment, promo, ad, or related content.

A candidate with fewer than roughly 300 visible characters should usually be rejected unless it is clearly the only article-like region.

The algorithm should prefer a container that includes the title and body together. If that is not available, it may select the body container and separately render `document.title` or the parsed title.

---

AG00 Implementation Detail: Tokenization

---

The tokenizer should not split text into individual characters by default. It should split into meaningful visual units.

For normal prose, a token can be a word plus following whitespace, punctuation, or a short phrase group. The renderer can group several tokens into a burst.

For headings, tokens should be phrase-sized.

For code, tokens should be lines or logical chunks.

For tables, tokens should be rows.

For captions, tokens should be shorter than normal prose, because captions are often compact and should feel deliberate.

Whitespace must be preserved where meaningful. The final rendered text should match the original visible text.

Inline elements should be reconstructed safely. For example, an anchor's text can stream inside an anchor element, but the `href` should be attached from the beginning or at block completion. The final anchor must be functional.

---

AH00 Implementation Detail: Renderer

---

The renderer should produce a final DOM that is structurally similar to the article content model.

For each block, the renderer creates the final element first, appends it to the replay surface, applies active styling, streams content into it, then marks it complete.

The renderer should use `requestAnimationFrame` for visual updates and `setTimeout` or promise delays for pacing. It should avoid one DOM write per character. Instead, it should append text bursts.

A rendering controller should expose `pause`, `resume`, `skip`, `restore`, and `setSpeed`.

When paused, the async loop should wait without losing its current block and token index.

When skipped, the controller should render all remaining blocks immediately and call the completion routine.

When restored, the controller should abort all pending work and put the page back.

AJ00 Controls Specification

---

The control panel should be compact and readable. It should appear only while the bookmarklet is active.

The panel should include these actions: Pause, Resume, Skip, Replay, Restore, and Speed.

The Speed control can cycle through Relaxed, Standard, and Fast.

The panel should show a status label: Preparing, Streaming, Paused, Complete, or Restored.

The controls should not use icons alone. Text labels are clearer and easier to use across arbitrary pages.

The controls should remain available even if the user scrolls. If the replay surface is in view, the controls can sit inside the surface. If the surface scrolls out of view, a fixed compact control pill can remain in the viewport.

AK00 Browser Compatibility Requirements

---

The bookmarklet should target current desktop versions of Chrome, Edge, Firefox, and Safari.

The installation page should work as a static local or hosted HTML file.

The bookmarklet should avoid dependencies on module loading at runtime because bookmarklets execute as plain JavaScript URLs.

The bundled code should avoid top-level `await`.

The runtime should avoid relying on APIs that are unavailable in common current browsers unless a fallback exists.

Clipboard copying on the installation page may require a secure context. If clipboard copying fails, the page should allow manual selection of the generated URL.

AL00 Performance Requirements

---

The bookmarklet should start visibly within approximately 500 milliseconds on normal article pages.

Extraction should avoid expensive repeated full-document scans where possible.

The renderer should avoid rebuilding the entire article DOM for every token. It should append only the current burst.

The animation should be cancellable.

The bookmarklet should handle articles of at least 10,000 words without freezing the page.

For very large documents, the renderer should use chunked processing so the page remains responsive.

AO00 Open Implementation Decisions

---

The first decision is whether to bundle a full readability library or implement a smaller extraction fallback. The full library likely improves extraction quality but increases bookmarklet size. A small custom extractor reduces size but may fail more often.

The second decision is whether the bookmarklet should replace the original article container or render an overlay. Replacing the article container feels more magical and integrated. Overlay mode is safer on complex pages. The recommended behavior is replacement first, overlay fallback second.

The third decision is how closely to preserve the original article's styling. Full preservation can be difficult because streaming wrappers alter the DOM. The recommended behavior is to preserve semantic structure and broad typography rather than exact styling.

The fourth decision is whether to store the original article by HTML string or by DOM node clone. HTML string restoration is simpler. DOM node clone restoration may preserve more state but is more complex. The first version can use HTML string restoration for the target container.

The fifth decision is the default speed. The recommended default is Standard mode with burst-based token rendering and automatic acceleration for long articles.

AP00 Final Task Definition

---

Build a Bun.js-powered bookmarklet project that produces a single readable JavaScript bundle exposing `BookmarkletMain`. Create a static installation page that loads this bundle, converts `BookmarkletMain.toString()` into a `javascript:` bookmarklet URL, and renders a draggable bookmarklet link.

When activated on a web page, the bookmarklet identifies the main readable article content, preserves the original DOM for restoration, replaces or overlays the article area with a controlled replay surface, and streams the same content back into view with polished AI-chat-style visual effects. The effect should include structured block rendering, token bursts, cursor behavior, subtle blur and fade transitions, active block highlighting, adaptive pacing, speed controls, pause/resume, skip, replay, and restore.

The final result should be a serious, reversible, privacy-preserving visual transformation that makes an existing article appear to be generated live by an AI-style response interface while keeping the original content intact and readable.
