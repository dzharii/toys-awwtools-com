A00 Appendix B: Codex Development Constitution and Coding Guidance

---

This appendix defines how Codex should approach implementation work for the Topic Research Notepad proof of concept. It is a development constitution rather than a product feature document. Its purpose is to clarify expectations around implementation freedom, code organization, documentation, build tooling, testing, dependency limits, and communication back to the project owner.

The application is a local-first, static web application that will be bundled and evaluated as a proof of concept before any decision is made about integrating it into the browser extension. The code should be practical, maintainable, and resilient enough for real local evaluation, but it should not become over-engineered in pursuit of theoretical completeness.

---

B00 Authority of the Specification

---

The specification documents are recommendations, not rigid law.

Codex has full freedom to make implementation decisions that differ from the written specification when those decisions produce clearer, safer, more maintainable, or more functional software. The specification describes the intended product behavior and preferred architecture, but the implementation may reveal better approaches.

Codex should preserve the core goals even when changing details: local-first storage, browser-only operation, reliable autosave, block-based research pages, safe paste handling, IndexedDB persistence, modular code, searchable local data, and a compact retro-styled user interface.

When Codex deviates from the specification in a meaningful way, it should document the deviation in a Markdown note inside the repository. The note should explain what changed, why the change was made, what tradeoff was accepted, and whether the change should be revisited later.

---

C00 Implementation Judgment

---

Codex should use engineering judgment rather than mechanically following every sentence of the specification.

If a proposed API is too complex for the proof of concept, Codex may simplify it. If a field name is awkward, Codex may rename it. If a Worker boundary becomes difficult during early implementation, Codex may create an adapter that works on the main thread first, provided the persistence API remains separable. If a feature creates disproportionate implementation cost, Codex may implement a smaller version and document the decision.

Working software is preferred over a perfect-looking architecture that does not run. Data safety is preferred over clever editing behavior. Clear code is preferred over dense abstractions. Explicit interfaces are preferred over hidden coupling.

---

D00 Runtime and Tooling

---

The project will use Bun.

Bun is the expected tool for development scripts, bundling, and tests. Codex should assume the project is bundled rather than relying on a pile of unprocessed browser scripts. The final output should still be deployable as a static website, suitable for GitHub Pages or equivalent static hosting.

Codex may create or update Bun-related project files such as `package.json`, `bun.lock` if already present or generated by the environment, build scripts, test scripts, and source directories. The build should produce static assets that can be served without a backend.

The implementation should use modern JavaScript. TypeScript is not required unless the project owner explicitly changes that decision. Type information should be provided through JSDoc.

---

E00 Dependency Policy

---

Codex is prohibited from installing new NPM modules.

The implementation must use only files, libraries, and vendored dependencies already present in the project folder. If Dexie 4.2.0 is provided locally, Codex may use it. If the retro CSS framework is provided locally, Codex may use it. Codex must not run package installation commands to add new dependencies.

If Codex believes a missing library would materially simplify the work, it should not install the library. It should write a Markdown note explaining the missing capability, the suggested library or utility, the reason it would help, and the fallback approach used instead.

This rule applies to runtime dependencies, development dependencies, testing libraries, formatters, DOM sanitizers, editor frameworks, drag-and-drop libraries, markdown libraries, and utility packages. The project should remain vendor-controlled.

---

F00 Code Organization

---

Codex should prefer modular code.

The implementation should be organized around clear responsibilities: application bootstrap, storage client, storage Worker or storage adapter, database schema, page and block models, editor rendering, sidebar rendering, paste normalization, search indexing, export, and tests.

The exact file structure may differ from the earlier recommendation, but the boundaries should remain understandable. Storage logic should not be scattered across UI event handlers. Paste sanitization should not be mixed directly into database code. Search indexing should be testable independently from rendering. Export logic should consume internal data structures rather than scraping the DOM.

A good module should have a clear purpose, small public surface area, and predictable inputs and outputs.

---

G00 JSDoc Type Policy

---

The project uses JavaScript with JSDoc type annotations.

Codex should document important public APIs with JSDoc. This includes storage client methods, Worker message payloads, database record shapes, block content shapes, paste conversion functions, search functions, export functions, and any module-level API that other modules call.

JSDoc should include type information in a TypeScript-like style where useful. The goal is to make the code easier to inspect, safer to refactor, and clearer for future work without converting the project to TypeScript.

Example style:

```js
/**
 * Updates one block record and refreshes its derived search index entry.
 *
 * @param {string} blockId - Stable block identifier.
 * @param {Partial<ResearchBlock>} patch - Fields to merge into the block.
 * @returns {Promise<ResearchBlock>} Updated block record.
 */
export async function updateBlock(blockId, patch) {
  // implementation
}
```

Record typedefs should be defined near the modules that own them or in a shared model file.

Example:

```js
/**
 * @typedef {Object} ResearchPage
 * @property {string} id
 * @property {string} title
 * @property {string} createdAt
 * @property {string} updatedAt
 * @property {number} sortOrder
 * @property {string|null} archivedAt
 * @property {Record<string, unknown>} metadata
 */
```

JSDoc should be meaningful. Do not add noisy comments that repeat the obvious. Public boundaries and non-obvious behavior should be documented well.

---

H00 Worker and Backend API Documentation

---

The browser Worker acts as the local backend for the application. Its message protocol must be documented clearly.

Codex should document Worker request and response envelopes, supported message types, expected payloads, return values, and error shapes. This documentation may live in code comments, JSDoc typedefs, and a Markdown file if needed.

The Worker API is important because it separates the editor UI from IndexedDB persistence. Future extension integration may reuse or replace this boundary. Therefore, the message protocol should not be implicit knowledge hidden inside a switch statement.

Each Worker message handler should have a clear description of what it does and what data it returns. For example, `createPage` should specify whether it creates an initial block, whether it updates settings, and whether it returns only the page or the page plus blocks.

Error codes should also be documented. The UI should be able to distinguish initialization failure, validation failure, read failure, write failure, and protocol mismatch.

---

I00 Commenting Style

---

Comments should explain intent, constraints, and non-obvious decisions.

Good comments belong near public APIs, protocol definitions, schema definitions, migration logic, sanitization rules, debounced persistence logic, and places where a simpler-looking implementation would be unsafe.

Avoid comments that merely narrate the next line of code. Prefer comments that explain why the code exists, what invariant it protects, or what future maintainer should not break.

For example, a comment near paste handling should explain that pasted HTML is treated as untrusted and must be converted into internal block data before rendering. A comment near debounced saves should explain why saves are per block rather than per page. A comment near search indexing should explain that the index is derived and rebuildable.

---

J00 Testing Policy

---

Testing should use Bun's test environment.

Codex should prefer unit tests. The first useful tests should cover pure logic modules: model factories, block normalization, search text extraction, paste conversion, Markdown export, sort order utilities, and storage-client debounce behavior if it can be tested without fragile timers.

Tests should avoid requiring a real browser DOM unless necessary. If a function is hard to test because it is tightly coupled to DOM rendering, Codex should consider extracting pure logic into a separate module.

IndexedDB and Worker tests may be harder in the Bun environment depending on available APIs. Codex should not spend excessive effort forcing brittle tests around browser-only APIs. Instead, it can test the pure database-adjacent helpers and document any manual testing required for IndexedDB behavior.

The test suite should be practical. A small set of meaningful tests is better than a large set of shallow tests.

---

K00 Manual Testing Notes

---

Because this is a browser application, some behavior will need manual validation.

Codex should document manual test steps in a Markdown file when features are difficult to validate through unit tests. Manual testing notes should cover startup, page creation, autosave after reload, paste cleanup, search, Markdown export, JSON backup, sidebar navigation, and error states where practical.

Manual test notes should be written as steps that the project owner can follow in the browser. They should also mention expected results.

This is especially important for IndexedDB persistence, Worker loading, clipboard paste behavior, and GitHub Pages deployment behavior.

---

L00 Markdown Communication Back to the Project Owner

---

Codex is encouraged to communicate implementation decisions through Markdown files in the repository.

Useful documents include `IMPLEMENTATION_NOTES.md`, `DECISIONS.md`, `KNOWN_LIMITATIONS.md`, `MANUAL_TESTING.md`, or similar names. Exact filenames are flexible.

When Codex makes a meaningful decision that deviates from the specification, it should document it. When Codex encounters a missing dependency that would have helped, it should document it. When Codex implements a workaround, it should document why. When Codex leaves a feature incomplete, it should document what remains.

These notes should be concise but specific. They should not become a dump of every small coding choice. They should preserve decisions that matter for future implementation work.

---

M00 Handling Missing Libraries

---

Codex must not install missing libraries. If a useful library is unavailable, Codex should build a smaller local implementation or reduce scope.

Examples: if no HTML sanitizer library is available, Codex should implement a conservative local sanitizer using DOM parsing and explicit allowlists. If no Markdown library is available, Codex should implement basic Markdown export for the supported block types. If no drag-and-drop library is available, Codex should use simple move-up and move-down controls or basic browser drag events.

When this creates a limitation, Codex should document it. The note should state what would improve if a dependency were allowed and what fallback was implemented.

---

N00 Build Output Expectations

---

The build should produce static files that can be hosted without a server-side runtime.

Codex should make the build output easy to identify, such as a `dist` directory. The output should include bundled JavaScript, CSS, vendored assets or references to vendored assets, and an `index.html` file.

The app should not require live API calls. It should not require an NPM dev server at runtime. A local dev server can be used for development, but the deployed app should be plain static assets.

If Worker scripts require special bundling, Codex should document how they are emitted and loaded. Worker path issues are a common source of static deployment bugs, so this area should be handled deliberately.

---

O00 Source Style

---

Use modern JavaScript syntax appropriate for current Chromium-based browsers.

Prefer `const` and `let` over `var`.

Prefer explicit functions with clear names over anonymous large inline callbacks.

Prefer small pure helper functions where they improve testability.

Prefer object literals and plain data for records crossing Worker boundaries.

Avoid global mutable state except in deliberate application state modules.

Avoid storing DOM nodes in persistent state.

Avoid broad implicit side effects during module import. Application startup should happen from a clear bootstrap function.

---

P00 UI Code Guidance

---

The UI should be implemented in a clear and inspectable way.

A full frontend framework is not required and should not be added as a dependency. The app can use plain DOM rendering, template functions, or a small local rendering pattern.

UI modules should render from application state and call explicit handlers. They should not directly perform IndexedDB writes. They should call the storage client or application controller.

The editor should render internal block records, not raw pasted HTML. All displayed content should come from trusted internal data structures.

The UI should remain compatible with the provided retro CSS framework. If the framework provides components or classes, use them consistently. Avoid inline style sprawl unless necessary for dynamic layout.

---

Q00 Persistence Code Guidance

---

Persistence code should be explicit and conservative.

Database schema should be centralized. Record creation should go through factory or normalization functions. Writes should validate important fields. Search index updates should happen close to the writes that require them.

Debounced save code should be understandable. It should be clear how pending saves are stored, how they are flushed, and how save failures are reported.

Worker communication should use request IDs and structured responses. Do not rely on message order alone.

When implementing autosave, make sure the UI does not claim data is saved before the Worker confirms that IndexedDB write completed.

---

R00 Paste Handling Guidance

---

Paste handling is a product-critical area and should be implemented defensively.

Pasted HTML must be treated as untrusted. It should be parsed in a detached document, sanitized, converted into supported block records, and rendered through normal block renderers.

Do not insert raw pasted HTML into the editor.

If conversion is uncertain, prefer plain text. The user can recover from plain readable text. The user cannot easily recover from poisoned editor markup.

Paste conversion should be covered by unit tests where possible. Tests should include paragraphs, headings, lists, tables, links, code blocks, and unsafe tags or attributes.

---

S00 Export Code Guidance

---

Export should be implemented from stored records, not from rendered DOM.

Markdown export should consume page and block data and produce readable Markdown. JSON backup export should include structured records and version metadata.

Export code should handle unknown block types safely. Unknown blocks should not disappear silently. For Markdown export, an unsupported block can be represented as a fenced JSON block or a clear placeholder plus raw data.

Export should flush pending writes before reading durable data.

---

T00 Error Handling Guidance

---

Errors should be visible and structured.

The Worker should return error codes and messages. The storage client should propagate them. The UI should show a compact user-facing message and keep developer details available in console logs or implementation notes.

Do not swallow IndexedDB errors silently.

Do not clear local editor state after a failed write.

Do not reset the database automatically after migration or initialization failure.

When the app is in a degraded mode, such as no IndexedDB or no Worker, the UI should state that clearly.

---

U00 Acceptance of Workarounds

---

Workarounds are acceptable when they keep the proof-of-concept moving.

A workaround should be local, documented, and not misleading. If a feature is simplified, the UI should not pretend it is more complete than it is. If a technical compromise is made, it should be recorded in a Markdown note when it affects future development.

For example, if Worker-based IndexedDB is deferred and a main-thread adapter is implemented first, that is acceptable if the storage API remains separable and the decision is documented. If drag-and-drop is replaced by move-up and move-down buttons in the first pass, that is acceptable if the core ordering feature works. If paste conversion supports only a conservative subset of HTML, that is acceptable because safety and predictability are more important than perfect fidelity.

---

V00 Performance Guidance

---

The main performance rule is to avoid rewriting the full page document during normal edits.

Store blocks separately. Save the changed block. Debounce text edits. Batch paste insertion. Batch reorder operations. Rebuild only affected search index entries.

Avoid expensive DOM rerenders during typing. The editor should not reconstruct the entire application shell on every keystroke if that creates visible lag.

Avoid storing large raw HTML payloads by default. Normalize content into compact internal block records.

Performance should be good enough for normal local research documents. Do not prematurely build complex virtualization or advanced caching unless actual behavior requires it.

---

W00 Security and Safety Guidance

---

The app is local, but it still handles untrusted content from the web.

Treat pasted HTML as untrusted.

Escape user-provided text when rendering.

Do not execute scripts from pasted content.

Do not preserve inline event handlers.

Do not render arbitrary HTML from IndexedDB.

Do not allow a malformed record to crash the whole app.

Do not add remote scripts or CDN dependencies.

Do not introduce network calls unless explicitly required by the project owner.

---

X00 Documentation to Preserve in Code

---

Codex should make sure the following areas are documented either in JSDoc or Markdown:

The IndexedDB schema and why pages and blocks are separate.

The Worker message protocol.

The autosave debounce strategy.

The search index as derived data.

The paste sanitization and normalization rules.

The export format and JSON backup metadata.

Known limitations of the proof-of-concept.

Manual test steps.

Meaningful deviations from the specification.

These documents and comments should help a future implementer understand the system without rereading the entire conversation.

---

Y00 Definition of Done for Codex Implementation Work

---

A coding task should be considered complete when the implemented behavior works in the browser, the build runs through Bun, relevant unit tests pass, and meaningful decisions or limitations are documented.

For a feature involving persistence, completion requires at least one reload test: create or edit data, reload the page, and confirm the data returns.

For a feature involving paste, completion requires testing that external styles and unsafe markup do not enter the rendered editor.

For a feature involving Worker messages, completion requires documented request and response shapes.

For a feature involving a deviation from the specification, completion requires a Markdown note explaining the deviation.

---

Z00 Final Instruction to Codex

---

Build the proof-of-concept as real software, not as a throwaway demo, but keep the implementation proportional to the evaluation stage.

Use Bun. Use modern JavaScript. Use JSDoc for important types and public APIs. Do not install dependencies. Prefer unit tests for pure logic. Keep modules clear. Document Worker messages and storage APIs carefully. Preserve user data. Make reasonable implementation decisions when the specification is incomplete, and document important deviations in Markdown.
