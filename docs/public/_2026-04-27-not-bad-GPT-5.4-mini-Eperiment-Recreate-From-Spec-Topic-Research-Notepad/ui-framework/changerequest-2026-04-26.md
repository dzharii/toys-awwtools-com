# Change Request: Split Pane Component for AWW Bookmarklet UI Framework

Date: 2026-04-26

## Summary

This change adds a reusable split-pane layout component to the AWW Bookmarklet UI Framework:

```html
<awwbookmarklet-split-pane
  direction="horizontal"
  value="280"
  min-start="180"
  min-end="420"
  aria-label="Resize page sidebar"
>
  <aside slot="start">...</aside>
  <main slot="end">...</main>
</awwbookmarklet-split-pane>
```

The component provides two slotted content regions separated by a draggable and keyboard-accessible splitter. It is intentionally generic and not tied to Topic Research Notepad, although that app is the first integration point.

## Context

Topic Research Notepad needed a resizable Pages sidebar next to the editor canvas. The existing app layout used a fixed CSS grid column, which made longer page titles and future page-tree affordances hard to use. The notepad should let users widen the navigation area while organizing research, then shrink it while writing.

This need is not notepad-specific. A resizable two-pane primitive is useful across the broader framework for tools such as:

- page tree plus editor
- source metadata plus document body
- browser preview plus notes
- screenshot preview plus controls
- command output plus configuration
- search results plus selected record

For that reason, the resize behavior was added to the framework as a web component rather than implemented as one-off application code.

## Attributes and Properties

### `direction`

Allowed values:

```txt
horizontal
vertical
```

Default:

```txt
horizontal
```

Horizontal means the start pane is left, the splitter is vertical, and the end pane is right.

Vertical means the start pane is top, the splitter is horizontal, and the end pane is bottom.

Invalid values normalize to `horizontal`.

### `value`

Start pane size in CSS pixels.

For `direction="horizontal"`, this is the start pane width.

For `direction="vertical"`, this is the start pane height.

The component exposes a JavaScript property:

```js
splitPane.value = 300;
console.log(splitPane.value);
```

User resizing updates the property and reflects the attribute.

### `min-start`

Minimum start pane size in CSS pixels.

Default:

```txt
160
```

### `min-end`

Minimum end pane size in CSS pixels.

Default:

```txt
240
```

### `max-start`

Optional maximum start pane size in CSS pixels.

If omitted, the maximum is derived from available space minus `min-end`.

### `disabled`

When present, resizing is disabled and the splitter is visually muted.

### `aria-label`

An app-provided label for the internal separator.

If omitted, the component uses:

```txt
Resize panels
```

## Slots

The component defines two named slots:

```html
<slot name="start"></slot>
<slot name="end"></slot>
```

The component does not assume any specific element type inside either slot.

Empty slots render safely and should not break sizing or pointer behavior.

## Events

### `awwbookmarklet-split-pane-resize`

Emitted during pointer drag and keyboard resize.

Detail:

```js
{
  value: 300,
  direction: "horizontal",
  dragging: true
}
```

For keyboard changes, `dragging` is `false`.

### `awwbookmarklet-split-pane-resize-commit`

Emitted when pointer resizing finishes and after keyboard resizing.

Detail:

```js
{
  value: 300,
  direction: "horizontal"
}
```

Applications should persist durable layout state from the commit event, not from every live resize event.

Example:

```js
splitPane.addEventListener("awwbookmarklet-split-pane-resize-commit", (event) => {
  storage.setSetting("sidebarWidth", event.detail.value);
});
```

## Layout Implementation

The component uses Shadow DOM and the same framework style pattern as other components:

```js
adoptStyles(shadow, [BASE_COMPONENT_STYLES, SPLIT_PANE_STYLES]);
```

Internal layout uses CSS Grid.

Horizontal mode:

```css
grid-template-columns: var(--_start-size, 280px) var(--_splitter-size, 8px) minmax(0, 1fr);
grid-template-rows: minmax(0, 1fr);
```

Vertical mode:

```css
grid-template-columns: minmax(0, 1fr);
grid-template-rows: var(--_start-size, 280px) var(--_splitter-size, 8px) minmax(0, 1fr);
```

Both pane wrappers set:

```css
min-width: 0;
min-height: 0;
overflow: auto;
```

This prevents common overflow bugs where slotted content forces the pane wider or taller than the split area.

The host fills the size provided by its parent:

```css
:host {
  display: block;
  width: 100%;
  height: 100%;
  min-width: 0;
  min-height: 0;
}
```

## Pointer Interaction

Pointer down on the splitter starts resizing.

The component stores:

```js
{
  pointerId,
  startClientX,
  startClientY,
  startValue
}
```

Horizontal resize uses:

```js
nextValue = startValue + (event.clientX - startClientX);
```

Vertical resize uses:

```js
nextValue = startValue + (event.clientY - startClientY);
```

The splitter uses pointer capture when available and listens for pointer move/up/cancel to finish cleanly.

## Keyboard Interaction

The splitter is focusable:

```html
role="separator"
tabindex="0"
```

Keyboard behavior:

- `ArrowLeft`: decrease start width in horizontal mode
- `ArrowRight`: increase start width in horizontal mode
- `ArrowUp`: decrease start height in vertical mode
- `ArrowDown`: increase start height in vertical mode
- `Home`: move to minimum
- `End`: move to maximum
- `Shift` plus arrow: use larger step

Step sizes:

- normal arrow: 10 px
- shift arrow: 50 px

Keyboard actions emit both resize and commit events.

## Accessibility

The internal splitter uses:

```html
role="separator"
aria-orientation="vertical|horizontal"
aria-valuemin="..."
aria-valuemax="..."
aria-valuenow="..."
aria-label="..."
```

ARIA orientation follows separator orientation:

- horizontal split uses a vertical separator, so `aria-orientation="vertical"`
- vertical split uses a horizontal separator, so `aria-orientation="horizontal"`

The component has a focus-visible style using framework focus ring tokens.

## Styling

The visual style follows existing framework tokens:

- `--awwbookmarklet-surface-inset-bg`
- `--awwbookmarklet-border-subtle`
- `--awwbookmarklet-focus-ring`
- `--awwbookmarklet-focus-ring-width`

The splitter is intentionally visible but restrained. It uses a small grip mark and cursor changes:

- horizontal split: `col-resize`
- vertical split: `row-resize`

No app-specific colors or notepad-specific selectors are used in the framework component.

## Clamping Rules

The split value is clamped with a pure helper:

```js
clampSplitValue(value, { available, minStart, minEnd, maxStart });
```

Rules:

- values are numeric and finite; invalid inputs fall back safely
- start pane should respect `min-start`
- end pane should respect `min-end`
- `max-start` limits the start pane when provided
- too-small containers degrade without throwing
- negative pane sizes are avoided

When the container is too small to satisfy both minimums, the helper favors a feasible non-negative value instead of throwing.

## Host Resize Handling

The component observes host size changes with `ResizeObserver` when available. When the host changes size, the current split value is reclamped.

If `ResizeObserver` is unavailable, the component falls back to window resize handling.

## Persistence Strategy

Persistence is intentionally not implemented inside the framework component.

The framework emits events. Applications decide whether and where to persist values.

This keeps the component reusable in contexts where persistence may live in:

- IndexedDB
- extension storage
- localStorage
- in-memory UI state
- URL state
- no durable storage

Topic Research Notepad persists `sidebarWidth` in its existing Worker-owned settings store.

## Manual Verification Expected

The following behavior should be manually verified in a browser before merging into the main framework codebase:

1. Horizontal split renders start and end slots.
2. Vertical split renders start and end slots.
3. Pointer dragging changes `value`.
4. Pointer release emits resize commit.
5. Arrow keys resize the splitter.
6. Home and End clamp to min/max.
7. Focus ring appears on the splitter.
8. Slotted content scrolls without forcing layout overflow.
9. Empty slots do not break the layout.
10. Disabled mode prevents resize.
11. Host resize reclamps oversized values.

## Known Limitations

The first version intentionally does not include:

- more than two panes
- pane tabs
- arbitrary docking
- collapsible panes
- double-click collapse behavior
- a custom splitter slot
- built-in persistence
- snap layouts
- app-specific sidebar behavior

These can be added later without changing the core two-pane API.

## Merge Notes for Main Framework Codebase

When merging into the main UI framework, review these points:

1. Confirm the `TAGS.splitPane` name matches framework naming conventions.
2. Confirm `AwwSplitPane` is the desired class name.
3. Decide whether the pure helpers should remain exported from the public bundle.
4. Move downstream tests into the framework test suite if available.
5. Add a split-pane demo to the framework catalog if desired.
6. Confirm generated `dist/` files are handled according to the main framework's release process.
7. Consider whether window resize fallback should remove its event listener with a stable bound handler. The current primary path is `ResizeObserver`; fallback behavior is acceptable for current use but can be tightened.
8. Consider adding a future `slot="splitter"` only if consumers need custom splitter content.
9. Keep persistence out of the framework component unless the main framework adopts a general layout persistence API.

## Contribution Rationale

This component fits the framework because it is a primitive layout control, not a notepad feature. It uses the framework's existing custom element, Shadow DOM, `adoptStyles`, token, and registration patterns. It avoids new dependencies and keeps application state ownership outside the framework.

The immediate result is a better Topic Research Notepad layout. The longer-term value is a reusable split-pane primitive for other AWW bookmarklet and extension tools.
