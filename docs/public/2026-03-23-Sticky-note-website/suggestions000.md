A00 Implementation brief for Codex

This document is a technical implementation brief addressed to OpenAI Codex, targeting a build-free, self-contained HTML page with CSS-heavy presentation and a light JavaScript enhancement layer. It is intended to sit above the two longer UX specifications and translate them into implementation guidance, decision constraints, and an internal verification checklist.

The most important constraint is this: the final product must remain an authored HTML document first, not an application framework project. The person maintaining the page will add and edit content directly in HTML. JavaScript exists only to enhance navigation, filtering, search, bookmarking, and expansion behavior. CSS carries most of the visual identity. The implementation should therefore optimize for clarity of structure, semantic DOM design, predictable styling, and maintainable progressive enhancement.

You should treat the two existing specifications as the product source of truth for visual UX and interactive UX. This brief exists to reduce ambiguity at implementation time and to steer choices toward the simplest robust solution that satisfies those specifications without introducing unnecessary complexity.

A01 Primary implementation objective

Implement a single-page Seattle reference board that feels like an organized board of sticky notes. The page must be self-contained, responsive, visually distinctive, readable, and maintainable. It should work as a static HTML document even before JavaScript runs. JavaScript should progressively add filtering, full-text search, favorites, and expansion controls.

The implementation should prioritize these properties in this order: correctness of authored content rendering, readability, responsive layout stability, semantic DOM structure, deterministic styling, and then feature richness.

Do not optimize for abstraction too early. Do not introduce a build step. Do not introduce a framework. Do not assume server-side rendering. Do not fetch content from external JSON. The HTML document is the content store.

A02 Recommended implementation architecture

Use plain HTML, plain CSS, and plain JavaScript. Keep the code organized but do not over-engineer it. The best shape is one HTML document, one CSS file or embedded style block, and one JavaScript file or embedded script block. If external files are used, they should still keep the project build-free and easy to open locally.

The DOM should be semantic and queryable. Categories should be sections. Notes should be articles. Headings should be real headings. Search, filter, and personalization logic should operate by reading metadata directly from note elements and category definitions already present in the DOM.

The preferred implementation model is progressive enhancement. The board and notes must render correctly with no script. Script should then scan the document, derive an in-memory representation of notes and metadata, attach event handlers, build the filter options, manage local storage, and apply state changes by mutating classes, attributes, and visibility on existing DOM nodes.

A03 What to avoid

Do not implement this as a React, Vue, Svelte, or similar application unless explicitly told to do so later. Do not introduce package managers, transpilers, bundlers, or CSS preprocessors. Do not make the content depend on runtime templating. Do not rely on layout techniques that break source order just to create a more chaotic board effect.

Do not use true randomness for note styling on each page load. The visual variation must be deterministic and repeatable. Do not use absolute positioning for the board layout unless a very small decorative layer requires it. The notes must remain in accessible reading order.

Do not hide content in ways that make it unavailable to the custom search logic. Do not persist too much state in a way that makes reload behavior surprising.

A04 Authoring contract

Assume the human maintainer will edit HTML directly. Design the document structure so that adding a new note is easy and low risk. A person should be able to duplicate an existing note block, change the GUID, title, text, category metadata, tags, and optional details content, and be done.

This means the markup contract needs to be explicit and stable. Every note must expose a GUID. Every note must expose a normalized category key. Optional tags should be representable in a normalized form. Expandable text should have a clearly designated region. The visible note title should be easy to extract for search results and navigation.

Avoid forcing the author to manually maintain derived UI such as search indexes, favorites lists, or active filter displays. Those should be generated by script from note metadata.

A05 Recommended DOM model

Use a centered outer page shell and then a board container. Inside the board, use category sections. Inside each category section, use a heading and a notes container. Inside the notes container, place note articles.

Each note article should expose at least the following machine-readable properties through attributes or equivalent stable selectors: a GUID identifier, a category key, optional tags, and possibly a size or density hint if the author chooses to provide one. The note should also contain a human-visible title element, a visible body content region, an optional links region, and an optional expandable details region.

The category system should not be inferred only from note text. Maintain a central authored category definition region or a simple constant in script so labels, order, and emoji stay controlled. Notes should reference category keys from that controlled set.

A06 Suggested metadata strategy

Use stable normalized keys for machine-readable metadata and keep display labels separate. For example, a category may display an emoji and a human label, but the note should carry a plain normalized key such as emergency, news, transit, food, or favorites. The same principle applies to tags if tags are later added.

For GUIDs, assume the author provides a unique value. The bookmark system must rely on that value as the persistent identity key in local storage. Do not derive persistent identity from DOM position, title text, or category membership.

If you need to support future extensibility, prefer data attributes for note metadata because they remain close to authored content and easy to inspect manually.

A07 Visual implementation guidance

The board must look tactile, slightly irregular, and CSS-driven. Achieve this with layered gradients, subtle shadows, controlled transforms, and note variants. The note styling must be deterministic. Assign variants by repeatable sequence or by authored class. A repeating nth-child pattern is acceptable as long as it remains visually controlled.

The board should not become a generic card grid. However, the layout should still be implemented with a responsive grid or similar structured flow so that reading order and responsiveness stay reliable.

Use a maximum width for the board container so the layout remains composed on ultra-wide displays. Let the board collapse to one note per row on small screens. Let it expand to multiple columns on larger screens, but do not let the column count explode on very wide displays.

Keep the content readable under all decorative treatments. Decorative realism must never compromise contrast, spacing, or link clarity.

A08 Typography and note density implementation guidance

Implement typography so that short notes can appear visually larger and long notes remain readable. Do not shrink text endlessly to fit content. There must be a lower bound for body size, approximately in the territory of 14px. Past that point, the note should grow rather than the text shrinking further.

The simplest implementation path is to support authored density classes or note size hints first. Later, if desired, add a lightweight script that inspects content length or rendered overflow and assigns density classes automatically.

Do not attempt an overcomplicated text-fitting algorithm in the first version. The goal is maintainable behavior, not perfect typographic packing.

A09 Interaction implementation guidance

The interactive layer should initialize from the existing DOM and local storage. On load, the script should gather note references and metadata, construct the control state, apply any persisted favorites or filters, and then bind user interactions.

Filtering should operate by toggling state classes or hidden status on note or category elements. Search should operate by evaluating note text content against the query and then applying match, non-match, and optional hidden states. Favorites should operate by GUID lookup and local storage persistence. Expansion should operate on designated expandable regions and maintain an accurate expanded or collapsed state.

Keep the state model simple and explicit. There should be one current state object in script representing search query, active category filters, favorites set, strict search visibility mode, and possibly expansion mode if needed.

A10 Category filter implementation guidance

Implement category filtering as a multi-select system. The user must be able to select multiple categories at once. The UI may be a custom expandable chooser, but the underlying logic should be plain and accessible.

The selected categories should be reflected in an active filters row or equivalent visible summary. Removing a filter should update the board immediately. Clearing all filters should restore the default view of all authored categories.

Search should operate within the currently filtered scope. Favorites-only mode should also define scope. That means the board first decides which notes are in scope through category and favorites filters, then search evaluates matches within that scoped set.

A11 Favorites implementation guidance

Each note should expose a bookmark toggle. That toggle should add or remove the note GUID from a favorites set stored in local storage. The favorites feature is personal state and must survive reloads.

Enforce a maximum of ten favorites. When the user attempts to exceed the limit, block the action and show a concise explanatory message. Do not silently fail. Do not use a blocking modal unless there is no other practical option.

Favorites should be represented as a pseudo-category or equivalent dedicated filter scope. They do not replace or mutate the authored category of a note.

A12 Search implementation guidance

Implement a custom full-text search that inspects all relevant note text, including collapsed details content. The browser's native find behavior is not sufficient for this requirement because collapsed details may not participate in the expected way.

The search should be note-level, not fragment-level. A note is either a match or not a match for the current query. The results-summary area should list matching notes as links back to those notes.

Default search behavior should dim non-matching notes rather than hide them. This preserves board context. Also implement an explicit search-only mode labeled in a precise way such as "Hide non-matching notes". This mode must reset when the query becomes empty or is explicitly cleared.

Keep the search algorithm simple unless the content volume proves otherwise. A normalized case-insensitive substring match across aggregated note text is enough for the first version. You do not need fuzzy ranking unless asked later.

A13 Expandable details implementation guidance

Some notes will have details-summary content or an equivalent expandable area. That content must remain in the DOM at all times so the custom search can inspect it. The collapsed state is visual, not structural.

Each note with expandable content should have its own toggle. Also provide global expand-all and collapse-all controls in the main control area. These global controls should operate on the current scope of visible notes rather than on content that is currently fully hidden by unrelated filters.

Do not make expansion state surprisingly persistent across reloads unless explicitly requested later. The safer default is to keep expansion session-local only.

A14 Local storage guidance

Persist bookmarks and filter selections. These are stable convenience preferences and match the specification. Be careful with search persistence. Persisting a stale query often creates confusing reload behavior, especially if the board loads already dimmed or half-hidden. The safer default is not to persist live search text unless there is a strong reason and the UI makes that state extremely obvious.

If local storage contains stale GUIDs or obsolete category keys, ignore them gracefully. Do not let stale storage break initialization. Optionally clean invalid entries when saving next state.

Treat local storage as a convenience layer only. The DOM remains the authoritative source for actual content.

A15 Accessibility guidance

The final implementation must remain keyboard operable. Custom controls must expose clear states. Focus styles must remain visible. Search results links must navigate predictably. Bookmark toggles, expand toggles, and category selections must not depend on hover.

The page should remain readable and navigable without JavaScript. With JavaScript unavailable, the board should still display as a normal document. Interactive controls may not function, but they should not break the page or obscure content.

Respect reduced-motion preferences. Any hover lift, transition, or search-related animation should be subtle and reducible.

A16 Recommended implementation sequence

Start by implementing the static semantic HTML board and category structure. Then implement the CSS system: board shell, note anatomy, note variants, responsive grid, typography, and note density classes. After that, add the base JavaScript state model and DOM scanning. Then add category filtering, then favorites, then full-text search and search-results summary, then expansion controls, and finally local storage persistence and state restoration.

Do not start with search or favorites before the semantic note model is stable. Those features depend on correct metadata and DOM structure.

A17 What Codex should infer carefully

You should infer a clean markup convention even if the source specifications are written in UX language rather than DOM language. The author cares about maintainability by hand, so optimize for readable HTML over clever abstraction.

You should infer that the page may grow in content over time, so the metadata model must be extensible. Even if only categories and favorites are needed now, leave room for future tag filtering without reworking the entire approach.

You should infer that the visual design is intentionally irregular but must still be governed by a system. This means deterministic variant assignment, limited palettes, limited size buckets, and limited transform intensity.

You should infer that the user values correctness of layout and readability more than flashy effects. The design can be distinctive, but it must still look controlled.

A18 What not to leave ambiguous in your implementation

Do not leave note identity ambiguous. Require a GUID. Do not leave search scope ambiguous. Search should work within the current filter scope. Do not leave reset behavior ambiguous. Clearing search must also clear any search-only visibility refinement such as hiding non-matches. Do not leave favorites behavior ambiguous. Favorites are personal local state with a hard maximum of ten.

Do not leave metadata ownership ambiguous. The HTML is the authored source. The script reads it. The script does not invent note content independently.

Do not leave board width behavior ambiguous. The board must be responsive but bounded.

A19 Self-verification checklist for Codex

1. Verify that the page opens as a normal static HTML document with no build step, no framework runtime requirement, and no dependency on external data files.

2. Verify that the board remains readable and visually coherent before JavaScript runs.

3. Verify that categories are represented semantically in HTML and that notes are represented as discrete semantic content units.

4. Verify that every note requires and exposes a stable GUID used by the favorites system.

5. Verify that category metadata is machine-readable and not inferred only from visible labels.

6. Verify that adding a new note requires only editing HTML and does not require updating a search index, favorites list, or generated data structure by hand.

7. Verify that the board is centered and width-bounded on large screens and collapses cleanly on small screens.

8. Verify that mobile renders one note per row or an equally readable single-column layout.

9. Verify that note transforms and shadows remain subtle enough that text is comfortable to read.

10. Verify that visual note variation is deterministic and repeatable across reloads.

11. Verify that the note color system preserves adequate contrast for all text and links.

12. Verify that long notes do not reduce body text below the minimum readable threshold and instead grow in footprint when necessary.

13. Verify that category filtering supports multiple selected categories at once.

14. Verify that the active filter state is explicitly visible and removable from the UI.

15. Verify that clearing filters restores the default board view.

16. Verify that each note exposes a bookmark control with persistent local storage behavior.

17. Verify that the favorites limit is enforced at ten items and that exceeding the limit produces a visible explanatory message.

18. Verify that favorites can be viewed as a distinct scoped view or pseudo-category without altering the note's authored category.

19. Verify that notes with expandable details can be expanded individually.

20. Verify that global expand-all and collapse-all controls operate on the current visible scope.

21. Verify that collapsed details content remains present in the DOM for search indexing.

22. Verify that custom full-text search matches note titles, visible body text, and collapsed expandable text.

23. Verify that search results are presented in a reserved summary area above the board.

24. Verify that each search result links to the corresponding note on the board.

25. Verify that default search behavior dims non-matching notes instead of hiding them.

26. Verify that an explicit search-only option exists to hide non-matching notes.

27. Verify that the hide-non-matching option resets automatically when the search query is cleared.

28. Verify that clearing the search removes all search-mode styling and restores the default board presentation.

29. Verify that search operates only within the current category and favorites filter scope.

30. Verify that persisted local storage state is applied gracefully and that stale GUIDs or obsolete filter keys do not break initialization.

31. Verify that all controls are operable by keyboard and expose understandable states.

32. Verify that focus indication remains visible and not purely decorative.

33. Verify that the page remains usable with reduced-motion preferences.

34. Verify that the board still degrades gracefully if local storage is unavailable.

35. Verify that no layout choice breaks source order or basic accessibility for the sake of visual chaos.

A20 Reflection prompts for Codex during implementation

Before finalizing, ask whether any part of the implementation accidentally turned the document into an application shell rather than an authored page.

Ask whether any feature requires the author to maintain duplicate information by hand. If yes, simplify it.

Ask whether any decorative effect weakened legibility, hit targets, focus visibility, or responsive stability. If yes, reduce the effect.

Ask whether any persisted state could surprise a returning user. If yes, either remove that persistence or make the state more explicit.

Ask whether any custom control reproduces native behavior poorly. If yes, simplify it or revert to a more standard interaction pattern.

Ask whether any part of the code assumes randomness where the specification calls for deterministic procedural variation. If yes, replace it with a stable rule-based assignment.

Ask whether any note content is being excluded from custom search due to the way expansion or visibility is implemented. If yes, fix the search text aggregation.

A21 Final implementation stance

Implement the simplest robust version that honors the specifications. Favor semantic structure, deterministic styling, and progressive enhancement over abstraction for its own sake. The correct solution is not the one with the most components or the most clever code. It is the one that leaves behind a stable, editable HTML document with a distinctive visual system and reliable interaction behavior.
