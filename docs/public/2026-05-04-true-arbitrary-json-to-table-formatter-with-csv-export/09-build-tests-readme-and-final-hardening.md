## Specification

We build static RetroOS-style **JSON Table Inspector**. User pastes arbitrary JSON/JSONL from jobs, APIs, logs, or batch outputs. App turns it into readable spreadsheet-like table so user can quickly find failures, missing fields, errors, suspicious rows, and export clean visible data. This chunk is final bounded step in larger 10-spec plan.

README must explain product value and usage clearly. Final review must find and fix high-importance and medium-importance issues. Logging, security, accessibility, CSV export, data fidelity, and RetroOS integration must be audited.

This chunk value: previous chunks built features. This chunk turns feature set into stable tool that another person can run, understand, trust, inspect, and maintain.

```markdown id="soqjpm"
# JSON Table Inspector

JSON Table Inspector is a local-first browser tool for turning messy JSON or JSONL output into a readable table. It is designed for the common debugging moment where another program gives you a large batch result, API response, test report, log stream, or job output, and the important question is simple: which rows failed, what changed, and what should I look at first?

Instead of reading nested objects by hand, you paste the data into the app. The app parses the input, detects the most likely list of records, flattens nested fields into columns, highlights likely failures, and gives you table controls for search, sorting, filtering, column visibility, row details, and CSV export.

The tool does not send your data to a server. It runs in the browser as a static app. Your pasted data stays local to the page, except for anything you explicitly copy or download.
```

## Why use it?

JSON is good for machines, but it is often slow for humans to scan. A batch job might contain hundreds of result objects, and the failed rows may be buried under nested `error`, `status`, or `message` fields. JSON Table Inspector gives you a first useful view without asking you to write a script.

```markdown id="q75q4g"
## What the app does

### Parses JSON and JSONL

The input pane accepts strict JSON and JSONL. If parsing fails, the app keeps the error visible and shows the line, column, nearby text, and a suggested fix. The error does not disappear until the input is fixed or cleared.

### Detects records

Many JSON responses are objects that contain arrays inside them. The app scans the parsed data and chooses the most likely row source. It explains the choice and lets you select a different source when the guess is not what you need.

### Flattens nested fields

Nested fields become table columns such as `user.name`, `error.code`, or `metrics.durationMs`. Arrays are summarized by default so the table stays compact. The original row JSON is still preserved and available in the row details panel.

### Flags likely failures

The app looks for common failure signals: `success=false`, `ok=false`, `status=failed`, non-empty `error` fields, error codes, exception fields, and message text that looks like a timeout or failure. These are heuristics, not a replacement for your domain knowledge, so every flagged row includes a reason.

### Exports the visible table

CSV and TSV export use the current visible rows and visible columns. If you filter to failures and hide columns, the export follows that view. CSV values are escaped for commas, quotes, and line breaks, and formula-like text is protected before opening in spreadsheet tools.
```

### README export behavior

Need precise.

```markdown id="tzsxq2"
## Export behavior

Export uses the current table view:

- current search query
- current health filter
- current sort order
- current visible columns

CSV includes a header row. The Health column is included by default so exported rows keep their failure/warning context. Missing and `null` values export as empty fields. Objects and arrays export as compact JSON strings.

CSV cells are quoted when needed and double quotes are escaped according to standard CSV rules. Text values that look like spreadsheet formulas are prefixed with `'` to reduce formula-injection risk when opening the file in spreadsheet applications.
```

### README limits

Honest limits.

```markdown id="vkyaw9"
## Limits

This is a browser-only inspection tool. It is built for quick analysis, not for editing large datasets permanently.

Current limits:

- JSON must be strict JSON. Comments, trailing commas, and JSON5 syntax are not supported.
- Arrays inside rows are summarized by default instead of exploded into multiple rows.
- Very large inputs may be sampled or rendered with row limits to keep the browser responsive.
- Failure detection is heuristic. The app explains why a row was flagged, but your domain rules may be different.
- CSV export is a view export. It follows visible rows and columns, not necessarily every field in the original JSON.
```

### README troubleshooting

Include:

```markdown id="iafie4"
## Troubleshooting

### “Invalid JSON” appears

Check the line and column shown in the error block. Common causes are trailing commas, single-quoted strings, unquoted object keys, `undefined`, or a missing closing bracket.

### The app picked the wrong row source

Use the row source selector in the Mapping / Options area. The detector makes a best guess, but nested JSON can contain several arrays that look like records.

### I do not see a field in the table

Open the column picker. The app hides constant, mostly empty, very long, or low-priority columns by default to keep the first table readable.

### Export did not include a column

CSV and TSV export only visible columns. Show the column first, then export again.

### A large file feels slow

Use search, filters, or hide columns to reduce the visible table. The app may render only the first set of matching rows while keeping the parsed data available.
```

````markdown id="q9s69s"
# JSON Table Inspector

JSON Table Inspector is a local-first browser tool for turning messy JSON or JSONL output into a readable table. It is designed for the common debugging moment where another program gives you a large batch result, API response, test report, log stream, or job output, and the important question is simple: which rows failed, what changed, and what should I look at first?

Instead of reading nested objects by hand, you paste the data into the app. The app parses the input, detects the most likely list of records, flattens nested fields into columns, highlights likely failures, and gives you table controls for search, sorting, filtering, column visibility, row details, and CSV export.

The tool does not send your data to a server. It runs in the browser as a static app. Your pasted data stays local to the page, except for anything you explicitly copy or download.

## Why use it?

JSON is good for machines, but it is often slow for humans to scan. A batch job might contain hundreds of result objects, and the failed rows may be buried under nested `error`, `status`, or `message` fields. JSON Table Inspector gives you a first useful view without asking you to write a script.

It can help you:

- find rows where `success` is `false`, `status` is `failed`, or an error field is present
- compare passed and failed rows in the same table
- inspect sparse or mixed-schema JSON where not every row has the same fields
- keep the original row JSON available while using a flattened table view
- copy or download the current visible table as CSV or TSV

## Quick start

1. Open the app in your browser.
2. Paste JSON or JSONL into the input pane.
3. Check the detected row source. The app may choose `results[]`, `items[]`, `data[]`, the root array, or `JSONL records`.
4. Inspect the table. Use the health filter to show likely failures or warnings.
5. Open a row to see the full original JSON and the reasons it was flagged.
6. Copy or download the visible table when you need to move the result into another tool.

## Supported input shapes

### Root array

```json
[
  { "id": "a", "success": true },
  { "id": "b", "success": false, "error": { "message": "Timed out" } }
]
````

The app treats each array item as a row.

### Object with a record array

```json
{
  "jobId": "batch-123",
  "results": [
    { "id": "a", "status": "complete" },
    { "id": "b", "status": "failed" }
  ]
}
```

The app looks for likely record arrays such as `results[]`, `items[]`, `records[]`, or `data[]`. If it guesses wrong, use the row source selector.

### JSONL

```jsonl
{"id":"a","level":"info","message":"started"}
{"id":"b","level":"error","message":"timeout"}
```

Each non-empty line becomes one row.

## What the app does

### Parses JSON and JSONL

The input pane accepts strict JSON and JSONL. If parsing fails, the app keeps the error visible and shows the line, column, nearby text, and a suggested fix. The error does not disappear until the input is fixed or cleared.

### Detects records

Many JSON responses are objects that contain arrays inside them. The app scans the parsed data and chooses the most likely row source. It explains the choice and lets you select a different source when the guess is not what you need.

### Flattens nested fields

Nested fields become table columns such as `user.name`, `error.code`, or `metrics.durationMs`. Arrays are summarized by default so the table stays compact. The original row JSON is still preserved and available in the row details panel.

### Flags likely failures

The app looks for common failure signals: `success=false`, `ok=false`, `status=failed`, non-empty `error` fields, error codes, exception fields, and message text that looks like a timeout or failure. These are heuristics, not a replacement for your domain knowledge, so every flagged row includes a reason.

### Lets you control the table

You can search visible cells, sort columns, filter by health, resize columns, change density, wrap long text, choose visible columns, and define highlight rules for values that matter in the current dataset.

### Exports the visible table

CSV and TSV export use the current visible rows and visible columns. If you filter to failures and hide columns, the export follows that view. CSV values are escaped for commas, quotes, and line breaks, and formula-like text is protected before opening in spreadsheet tools.

## Export behavior

Export uses the current table view:

* current search query
* current health filter
* current sort order
* current visible columns

CSV includes a header row. The Health column is included by default so exported rows keep their failure/warning context. Missing and `null` values export as empty fields. Objects and arrays export as compact JSON strings.

CSV cells are quoted when needed and double quotes are escaped according to standard CSV rules. Text values that look like spreadsheet formulas are prefixed with `'` to reduce formula-injection risk when opening the file in spreadsheet applications.

## Privacy and data handling

The app is static and local-first. It does not upload pasted JSON to a server and does not fetch remote services to process your data. Browser features such as localStorage may remember UI preferences and, depending on the current app setting, the last input text. Use Clear when you want to remove the current input from the page.

## Limits

This is a browser-only inspection tool. It is built for quick analysis, not for editing large datasets permanently.

Current limits:

* JSON must be strict JSON. Comments, trailing commas, and JSON5 syntax are not supported.
* Arrays inside rows are summarized by default instead of exploded into multiple rows.
* Very large inputs may be sampled or rendered with row limits to keep the browser responsive.
* Failure detection is heuristic. The app explains why a row was flagged, but your domain rules may be different.
* CSV export is a view export. It follows visible rows and columns, not necessarily every field in the original JSON.

## Troubleshooting

### “Invalid JSON” appears

Check the line and column shown in the error block. Common causes are trailing commas, single-quoted strings, unquoted object keys, `undefined`, or a missing closing bracket.

### The app picked the wrong row source

Use the row source selector in the Mapping / Options area. The detector makes a best guess, but nested JSON can contain several arrays that look like records.

### I do not see a field in the table

Open the column picker. The app hides constant, mostly empty, very long, or low-priority columns by default to keep the first table readable.

### Export did not include a column

CSV and TSV export only visible columns. Show the column first, then export again.

### A large file feels slow

Use search, filters, or hide columns to reduce the visible table. The app may render only the first set of matching rows while keeping the parsed data available.
