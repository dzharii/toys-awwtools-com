# 2026-04-25 UI Review

## Part One: Design Diagnosis

The current interface has been converted from a warm, decorative card layout into a compact desktop utility surface based on the bookmarklet UI library. The visual thesis is now clear: a dense evaluator workbench with explicit borders, square controls, restrained status color, and tool-like panels. The largest remaining risk is not visual decoration, but component integration: custom-element controls must preserve native accessibility semantics, and library tabs must remain usable when labels overflow on small screens.

## Part Two: Summary of Problems

- ACCESSIBILITY: Library form controls do not forward ARIA label and description attributes from `awwbookmarklet-field` to their internal native controls.
- UX: Long tab labels in the examples panel can overflow horizontally on narrow screens because the library tablist is a fixed flex row without wrapping or scrolling.
- UX: The time zone field uses `awwbookmarklet-input` with `list`; browser datalist behavior may be inconsistent across Shadow DOM boundaries.
- SYSTEMATIZE: The custom code editor is token-aligned, but it is still a local component rather than a reusable `awwbookmarklet-*` component.
- SIMPLIFY: The context panel is dense and coherent, but the metric card carries exact timestamp text in its description, which can still wrap awkwardly.
- INVESTIGATE: Error-state keyboard behavior should be checked in a real browser because the error message focus button is a native button styled to match the kit rather than a library button.

## Part Three: Detailed Review

### ACCESSIBILITY: ARIA Forwarding In Library Controls

What is wrong: `awwbookmarklet-field` assigns `aria-labelledby`, `aria-describedby`, and `aria-invalid` to slotted custom elements, but the focusable controls are native inputs/selects inside Shadow DOM. The current input/select/textarea components do not forward those ARIA attributes to the internal native control.

Why it matters: Screen readers may focus an unlabeled inner control even though the host element appears labelled in the light DOM.

Future problem: Every downstream app using the field/control pairing inherits this issue. As more forms are built on the kit, accessibility debt spreads across the design system.

Fix type: High priority library fix.

Concrete change: Add `aria-label`, `aria-labelledby`, `aria-describedby`, and `aria-invalid` to mirrored attributes for `awwbookmarklet-input`, `awwbookmarklet-textarea`, and `awwbookmarklet-select`. Checkbox/radio should also mirror description and invalid state where useful.

### UX: Example Tabs Overflow

What is wrong: The examples use several long labels: "Business day shifting", "Formatting transforms", and "Holidays and observances". The library tablist is a single flex row with no wrapping or horizontal scrolling.

Why it matters: On mobile, tabs can push the page wider than the viewport or hide categories off-screen without an obvious affordance.

Future problem: More categories or localized labels will make the tab system break earlier.

Fix type: High priority library fix.

Concrete change: Update `awwbookmarklet-tabs` so `#tablist` supports `overflow-x: auto`, `max-width: 100%`, and stable whitespace. Keep keyboard behavior unchanged.

### UX: Time Zone Datalist In Shadow DOM

What is wrong: The time zone input mirrors the `list` attribute, but browser support for datalist lookup across Shadow DOM boundaries can be uneven.

Why it matters: Suggestions are a meaningful interaction surface for a time zone field. If suggestions do not appear, users must remember exact IANA IDs.

Future problem: If the zone list expands, users will expect typeahead selection, not just validation after typing.

Fix type: Medium priority replacement.

Concrete change: Convert this to a library-native combo/listbox component later, or pair the text input with a library select for common zones. For this pass, keep the text input because arbitrary IANA IDs remain important.

### SYSTEMATIZE: Code Editor Is Local

What is wrong: The editor now matches the library visually, but it is still a local native textarea/highlight overlay rather than a registered `awwbookmarklet-code-editor` component.

Why it matters: The editor is the primary interaction surface. Keeping it local is acceptable for this app, but it cannot be reused by other bookmarklet-library tools.

Future problem: Adding gutter markers, multiline diagnostics, or readonly modes will grow app CSS rather than library primitives.

Fix type: Medium priority component extraction.

Concrete change: Extract the editor shell into a new `awwbookmarklet-code-editor` if another app needs the same behavior.

### SIMPLIFY: Effective-Now Metric Text

What is wrong: The metric now uses a compact value, but the exact timestamp in the description can still wrap in tight containers.

Why it matters: This is secondary context, so it should not compete with the main controls or cause a visually noisy card.

Future problem: Longer zone names or localized date formats will worsen wrapping.

Fix type: Low priority simplification.

Concrete change: Keep the compact value as the main display and move exact timestamp to a tooltip or developer details if the card becomes cramped.

### INVESTIGATE: Native Error Button

What is wrong: The error message action is a native button styled with tokens inside a library alert.

Why it matters: It looks consistent now, but it does not inherit library button behavior such as busy/pressed states.

Future problem: If the error action gains menus, icons, or command wiring, this one-off button becomes inconsistent.

Fix type: Low priority replacement.

Concrete change: Replace with `awwbookmarklet-button` if the library supports the needed full-width, text-heavy action pattern.

## Part Four: Stronger Design Directions

### Direction 1: Dense Evaluator Workbench

Design thesis: Treat the app like a debugger for date expressions.

Typography direction: System UI at compact sizes, monospace only for expressions, output, and diagnostics.

Color and contrast direction: Neutral gray surfaces, blue for focus/selection, green/red/yellow only for semantic states.

Spacing and layout rhythm: Tight 4/8/12px rhythm with panels packed for scanning.

Component shape language: Square controls, explicit borders, inset code panes.

What to avoid: Hero layouts, decorative gradients, large rounded cards, and centered marketing copy.

Why this fits: The user is evaluating expressions and reading diagnostics; speed and trust matter more than ambience.

### Direction 2: Split Console

Design thesis: Make the editor and result feel like a two-pane console.

Typography direction: Slightly stronger monospace presence, with small system labels around it.

Color and contrast direction: Muted light shell with darker code/result panes.

Spacing and layout rhythm: Larger editor/result panes, fewer surrounding panels, examples collapsed behind tabs.

Component shape language: Terminal-like rectangles using library borders and status bars.

What to avoid: Dashboard-style metric grids.

Why this fits: The app's primary loop is type, evaluate, inspect.

### Direction 3: Reference Browser

Design thesis: Emphasize examples and fixtures as a browsable language reference.

Typography direction: Compact UI labels with monospace expression rows.

Color and contrast direction: Neutral library palette with selected examples highlighted by selection color.

Spacing and layout rhythm: Left-side example list, right-side editor/result stack.

Component shape language: Library listbox/list-item patterns and tabbed reference panels.

What to avoid: Cards for every category.

Why this fits: New users learn the date language by selecting examples and seeing diagnostics.
